# Specific Phase 0 + Phase 1 Implementation Plan

Code-specific, step-by-step build plan for turning the rust-code-mcp workspace
into an RL environment + reward signal. Generated by a 20-agent workflow (one
agent per step, each grounded in the actual source). Companion to the design
docs: `phase-1-plan-v3.md` (design), `phase-1-implementation.md` (steps +
decisions D1–D4), `m0-spikes.md` (feasibility gates).

Each step section below lists: goal, new/modified crates, a files table
(path · op · purpose · LOC prod · LOC test), new/modified/deleted types and
functions with real signatures + locations, key algorithm, dependencies, and
risks. Names follow a shared architecture contract (canonical crate layout, the
5-verb API, D1–D4). Anything an agent could not verify against source is marked
`(verify)`.

## Canonical Resolutions (authoritative — supersedes any conflicting step text)

The 20 step sections were drafted by independent parallel agents with no shared
memory; they collide on names, owners, and a few whole subsystems. This section
is the single source of truth. **Where a step section below contradicts this,
this section wins.** It folds in the doc's original 8-item list, the deeper
duplications found in review, and the borrow-ownership constraint.

### A. Action layer — one owner (P1.X), one name set
P1.8 and P1.X independently defined the *entire* action layer in the same file.
**P1.X owns it; P1.8 consumes it.**
- **Owner P1.X** (`rmc-env/src/action.rs`): `Action` (5 variants
  `Navigate`/`Analyze`/`Query`/`Crud`/`Meta`), the sub-enums, `ActionCategory`,
  `Observation`, `EnvSession`, `Action::dispatch(self, &mut EnvSession) ->
  Observation`.
- **P1.8 owns only**: `EpisodeRunner`, `Trajectory`/`TrajectoryStep`/`StepRecord`,
  `Policy`/`ScriptedPolicy`/`ApiPolicy`, `StepBudget`/`Termination`/
  `EpisodeOutcome`, `Episode`/`TaskSpec`. It does **not** define `Action`, the
  sub-enums, or the per-episode aggregate.
- **Renames (P1.8's names are dropped):** `AnalyzeOp/QueryOp/MetaOp` →
  `Analyze/Query/Meta`; `ActionKind` → `ActionCategory`; `StepResult` →
  `Observation`; `EpisodeState` → `EnvSession`.

### B. MCP env surface — session API canonical
P1.8's blocking `run_episode` and P1.X's session API are two designs; the
session API is canonical because RL needs interactive step-observe-decide.
- **Canonical (P1.X):** `env_open / env_step / env_view / env_reset /
  env_simulate / env_reward` over `EnvRegistry`; one `tools/env/` module, one
  `tools/params/env.rs`.
- **`run_episode` (P1.8) becomes a thin server-side driver** that opens a
  session and steps a `ScriptedPolicy`/`ApiPolicy` to completion over `env_step`
  — a convenience wrapper, not a second API. It lives in `tools/env/`. Drop the
  separate `tools/endpoints/env.rs` and the duplicate `RunEpisodeParams`.

### C. Shared reward / gate / audit model — one module
P1.4 (sim), P1.6 (gate), P1.7 (reward) each redefined the same types. **New
module `rmc-env::reward_model` owns them; all three import, none redefine:**
`Severity`, `GateOutcome`, `PrincipleViolation` (merge P1.6's `Penalty` in —
drop `Penalty`), `AuditSnapshot`, `AuditDelta`, `GraphMetrics`,
`GraphMetricsDelta` (merge P1.4's `MetricDelta` in — drop `MetricDelta`).
Plus **`rmc-env::audit_battery::run_audit_battery(loaded, snap, scope) ->
AuditSnapshot`** — the single orchestrator of the audit set
(`recursion_check`/`dead_pub_report`/`unsafe_audit`/`fn_body_audit`/
`crate_dependency_metric`). P1.6/P1.7/P1.4 call it; none re-batch the audits.

### D. Shared graph helpers — adjacency + cycle detection (in rmc-graph)
- **Cycle detection — one impl.** `rmc-graph` exposes
  `OpenedSnapshot::would_create_crate_cycle(consumer, producer) -> bool` (SCC
  over `crate_edges`, reusing `recursion_check`'s machinery). P1.5b, P1.5e, P1.4
  all call it. Drop the three ad-hoc cycle checks.
- **Adjacency — one set of builders, in rmc-graph:** `crate_edge_graph()`,
  `module_edge_pairs()`, `call_adjacency()`. Consumed by rmc-analyze (P1.3a
  spectral, P1.3b affinity) and rmc-env::reward (P1.7). P1.3a drops its own
  `structural_adjacency`; P1.4/P1.7 build no private adjacencies.

### E. EditPlan / compute_effects / persist — owner P1.5a
The apply pipeline split is **owned by P1.5a** (`rmc-env::crud`): `EditPlan`,
`compute_effects(&self, ...) -> EditPlan`, `persist(&mut self, EditPlan)`.
P1.5a's file list gains `crud/effects.rs`. **P1.4's simulator is a dry-run mode
that calls `compute_effects` and skips `persist`** (it does not introduce the
split). P1.5d consumes. This removes the build-order inversion (P1.5a precedes
P1.4 in M2a→M2b).

### F. CRUD module / engine / errors / cycle-refusal
- **File:** `rmc-env::crud` lives in `crud.rs` (file-based; submodules under
  `crud/`). Drop `crud/mod.rs` and `crud/engine.rs`.
- **Engine:** `CrudEngine` only. Drop the phantom `Editor`.
- **Errors:** `rmc-graph::graph::patch::PatchError` → `rmc-host::ApplyError`
  (`#[from] PatchError`) → `rmc-env::crud::error::CrudError` (`#[from]
  ApplyError`). Drop `GraphError`. CRUD error module = `rmc-env::crud::error`.
- **Cycle refusal:** `RefusalReason::Cycle` only (P1.6). Drop the `CycleRefusal`
  type; the cycle check returns a bool and the CRUD op constructs the `Refusal`.

### G. Host / storage ownership
- **`WorkingSnapshot` defined once** in `rmc-graph::graph::working` (the storage
  primitive: mutable LMDB copy + `compute_patch`/`apply_patch`), with
  `WorkingSnapshotId` (D1). **rmc-host does NOT redefine it**; `WorkspaceHost`
  holds one as a field. (Resolves the cross-crate name clash.)
- **`UndoLog`/`UndoEntry` defined once** in `rmc-host::undo` (P0.3, the complete
  D4 form). P0.2c's `apply_patch` *records into* it; P0.2c does not define its
  own. **(was build blocker)**
- **Host apply entrypoint = `WorkspaceHost::apply_edits`** (P0.2a); `ApplyError`
  lives in `rmc-host::error`. Drop `WorkspaceHost::apply`.
- **`EditClass` owner = P0.2b** (`rmc-graph::graph::affected`, next to
  `affected_set`); rmc-host imports it.

### H. Build-order & pinned signatures
- **`SCHEMA_VERSION`:** P0.1 → **13** (adds `seed`/determinism); P1.2 → **14**
  (adds `descriptions_by_target` + the `DEFAULT_MAX_DBS` raise). Sequenced, no
  double-bump. **(was build blocker)**
- **`emit_crate_scoped` owner P0.2b** (`rmc-graph::graph::extract`). Pinned sig:
  `pub fn emit_crate_scoped(loaded: &LoadedWorkspace, krate: Crate, model: &mut
  ExtractionModel) -> Result<()>` (verify against `emit_crate`). All others
  consume.
- **`apply_source_change` owner P1.5b** (`rmc-server::semantic::rename`). Pinned
  sig: `pub fn apply_source_change(host: &mut WorkspaceHost, change:
  SourceChange) -> Result<Vec<PathBuf>, ApplyError>` (applies RA `SourceChange`
  to the warm-host VFS, returns changed files). P1.5c reuses this exact sig.
- **`fn_body_range` owner P1.5a** (`rmc-graph::graph::skeleton::source`); P1.5d
  reuses. `find_item_syntax` visibility raised once (by P1.5a).

### I. Workspace dependencies — declared once
`petgraph`, `rand`, `rand_chacha`, `toml_edit`, `fs_extra` declared once in root
`Cargo.toml` `[workspace.dependencies]` (**owner P1.X**); all crates inherit.

### J. Warm-host borrow/ownership (architecture constraint, not a rename)
`CrudEngine` needs `&mut WorkspaceHost` and `Simulator` needs `&WorkspaceHost`;
holding both simultaneously inside `EnvSession` is an aliasing-`&mut` error that
may not compile. **Resolution:** `EnvSession` owns `WorkspaceHost` **by value**;
`CrudEngine`/`Simulator` are *not* stored holding a borrow — `dispatch`
constructs them per-arm with a fresh reborrow of the host. If parallel rollouts
later force it, move the host behind the P0.2a mpsc channel-actor. This
constraint may reshape the `dispatch` signature and must be settled before P1.8.

### K. DRY notes (non-blocking, fold during implementation)
- `description_cache.rs` (P1.2) hand-clones `embedding_cache.rs` — instead
  extract a generic `ContentHashCache<R>` in `rmc-graph` and instantiate it for
  both embedding and description records.
- The thin rmc-graph scan helpers (`reverse_crate_edges`, `who_uses_targets`,
  `callees_adjacency`, `references_in_span`, `crate_edge_graph`,
  `module_edge_pairs`) each re-walk `usages_by_id`/`crate_edges` — build them on
  one shared scan pass in `rmc-graph::query`.

### Priority order for resolution
1. **A + B** (action layer + MCP surface) — the largest structural collision.
2. **G `UndoLog` + H `SCHEMA_VERSION`** — the two build blockers.
3. **C** (shared reward_model + audit_battery) — prevents silent type divergence.
4. **J** (host borrow constraint) — may not compile otherwise.
5. **D, E, F, I, K** — name/owner cleanups, mechanical once 1–4 are set.

**Functions / methods** (new/mod/del):
- `fn extract(loaded: &LoadedWorkspace) -> ExtractionModel` @ rmc-graph::graph::extract — mod: collect `def_to_node` into a `Vec`, `sort_by_key(|(_, node_id)| *node_id.as_bytes())` before `extract_usages`/`extract_signatures`/`extract_statics` so their Vec push-order is deterministic; call `model.sort_canonical()` before returning.
- `fn extract_usages(model, db, vfs, def_to_node: &[(ModuleDefId, NodeId)], module_node_for)` @ rmc-graph::graph::usages — mod: change the `def_to_node` parameter from `&HashMap<..>` to a pre-sorted slice (or iterate a sorted key list) so the `for (&def_id, &target_node_id)` loop is order-stable; per-item `results.references` already sort by file then iterate refs in source order (verify ref-within-file order is positional).
- `fn extract_bindings(...) -> HashMap<ModuleDefId, NodeId>` @ rmc-graph::graph::bindings — mod: seed loop `for (&module_id, &node_id) in module_node_for` → iterate `module_node_for` sorted by `node_id` bytes so the seeded `def_to_node` entries (and any incidental ordering) are stable; `def_map.modules()` order is RA-internal — push results then rely on the final `sort_canonical`.
- `fn sort_canonical(&mut self)` @ rmc-graph::graph::model::ExtractionModel — new: stable-sort `bindings` by `(from_module bytes, namespace, visible_name, target bytes, kind)`, `usages` by `(target bytes, file, start, end)`, `contains` by `(parent bytes, child bytes)`, `signatures`/`statics` by `target bytes`. `nodes` is already a `BTreeMap` (deterministic) — left as-is.
- `fn canonical_dump(snap: &OpenedSnapshot) -> Result<CanonicalDump>` @ rmc-graph::graph::determinism — new: iterate every sub-DB via a RoTxn, hash each value with SHA-256, push `(table_name, key, value_hash)`, sort the whole vector; the golden test compares two dumps for equality. (DUP_SORT tables compared as sorted key→value-bytes sets.)
- `fn write_model(...)` @ rmc-graph::graph::snapshot — mod: also `meta_by_key.put("seed", &seed.0.to_le_bytes())`; signature gains `seed: GlobalSeed`.

**Key logic / algorithm:**
- IDs are already content-addressed and order-independent: `NodeId::from_components` (ids.rs) SHA-256s the component tuple, so the *set* of nodes/bindings/usages is reproducible for free. The defect is purely **iteration/insertion order** into the `Vec` fields of `ExtractionModel` (model.rs: `bindings`/`usages`/`contains`/`signatures`/`statics` are `Vec`; `nodes` is a `BTreeMap` and already fine).
- Primary nondeterminism source: `extract_usages` (usages.rs:45) iterates `for (&def_id, &target_node_id) in def_to_node` where `def_to_node: HashMap<ModuleDefId, NodeId>` — HashMap order is randomized per process, so `model.usages` (and `signatures`/`statics`, which also consume `def_to_node`) get nondeterministic push-order. Fix = iterate `def_to_node` in NodeId-byte order; belt-and-suspenders = `sort_canonical()` after extract.
- Secondary source: `extract_bindings` (bindings.rs:42) seeds from `module_node_for: HashMap`, and `emit_crate`/`extract_bindings` both walk `def_map.modules()` (RA-internal order, not guaranteed stable across loads). Final `sort_canonical()` normalizes the emitted `bindings`/`contains` regardless of walk order.
- `write_model` (snapshot.rs:407+) writes `nodes` from the BTreeMap and the Vecs in stored order, so once the model Vecs are canonicalized, `bindings_by_id`/`usages_by_id` primary keys (content-addressed via `binding_id_for`/`usage_id_for`) and the DUP_SORT secondaries (LMDB sorts duplicates internally) are reproducible.
- Seed: thread `GlobalSeed` through `BuildOptions` → `write_model` → `GraphManifest.seed` + `meta_by_key["seed"]`; `SCHEMA_VERSION` bumps 12→13 (graph_id_for hashes SCHEMA_VERSION, so old snapshots auto-rebuild — same pattern documented for v2..v12 in storage.rs).
- Golden test: `build_and_persist` twice into two `data_dir_override` temp dirs (force_rebuild), open both, assert `canonical_dump(a) == canonical_dump(b)`; a second in-process double-build catches HashMap-seed-dependent ordering that a single process might mask.

**Depends on:** [] (foundation step; nothing precedes it).
**LOC:** prod +195 / test +130.

**Risks / unknowns:**
- `def_map.modules()` ordering across RA reloads is RA-internal; `sort_canonical()` covers `bindings`/`contains` but verify no other pass leaks RA iteration order into a persisted Vec (impls.rs `extract_impl_items` mutates `def_to_node` and `model.nodes`/`contains` — nodes via BTreeMap are safe, but its `contains`/binding pushes must also be caught by the final sort). (verify)
- `usages_by_id` keys are `usage_id_for(usage)` — confirm that function hashes only content fields (target/file/start/end/category), not insertion index, else the *key set* (not just order) drifts. (verify the exact fields of `usage_id_for` / `binding_id_for` in snapshot.rs — not yet read.)
- DUP_SORT value bytes are LMDB-sorted, so secondary-index order is already deterministic; the golden dump must compare them as sets, not insertion sequences, to avoid a false negative.
- SCHEMA_VERSION bump invalidates all existing on-disk snapshots (intended) — coordinate the single bump with any other M0 step that also bumps it to avoid a double increment.
- Right-sized-rigor scope: this guarantees cold-build reproducibility only. P0.2's warm-host incremental path must separately match these canonical orderings on the reward-bearing fields (cross-check via `canonical_dump` after each incremental patch); fully matching DUP_SORT ordering incrementally is out of scope here (issue #11 in the register).

This step stops at producing the scoped model + the affected-set; it does not write LMDB.
- **No-textual-diff classifier.** Per D2, `EditDescriptor.class` is supplied by the CRUD op (it knows it changed only a body vs a signature). `classify()` never re-derives the class from source, avoiding a fragile diff parser.

- **Ownership (consumes, does not define):** `WorkingSnapshot`/`WorkingSnapshotId` are defined ONCE here in `rmc-graph::graph::working`; `rmc-host`'s `WorkspaceHost` (P0.2a) merely HOLDS a `WorkingSnapshot` field (a use, not a def). `UndoLog`/`UndoEntry` are consumed from `rmc-host::undo` (P0.3) — recorded into here, never defined here. `EditClass` (P0.2a) and `AffectedSet` (P0.2b) are likewise consumed.
- **Init (D1):** at episode start, host calls `WorkingSnapshot::init_from_base`, which `mdb_copy`s the published snapshot's `data.mdb` (heed `Env::copy_to_file`, the safe wrapper around `mdb_env_copy`) into `working/<session_id>/`, then opens it RW. Identity is `(session_id, base_graph_id, edit_seq=0)` — never `graph_id_for(...)`, so byte-flips don't churn the dir. One copy amortized over ~50 steps (right-sized rigor).
- **compute_patch as a model-vs-LMDB diff:** the after-edit `ExtractionModel` comes from `emit_crate_scoped` (P0.2b) restricted to `affected`. Because IDs are content-addressed (`NodeId::from_components`, `binding_id_for`, `usage_id_for`), a key is "changed" iff its value bytes differ and "removed" iff present in base but absent in the scoped model for an affected crate. This is exactly the contract the existing `write_model` loop encodes (snapshot.rs §1-5), run as a delta instead of a full write.
- **D3 matrix, column by `EditClass`:** Body → only `Usages`/`usages_by_consumer_function` patched + `signatures`/`statics` *unchanged* (matrix says `—`/`D-if-static`) + `meta` usage_count P; Sig/vis → `nodes` P, `bindings`(+both secondaries) P, `signatures` D, `statics` D; Item± → adds `children_by_parent` (contains) P; Mod-tree → contains + import bindings P, signatures/statics `—`; Macro/Cargo → `FullRebuild` (re-run `write_model` into the working env, matrix column `F`). Embeddings/descriptions are content-hash caches (`C`) — never touched here; they self-invalidate via `EmbeddingRecord.content_hash`.
- **DUP_SORT correctness (issue #5):** every secondary is `(NodeId→Id)` DUP_SORT. Deletes use `delete_one_duplicate(key, value)` (not `delete(key)`, which would nuke all dups for that key); inserts re-`put` the precise pair. `secondary_keys_for_*` regenerates the full secondary set for each changed/removed primary so primary and secondaries stay coherent — the silent-corruption failure mode the plan flags.
- **Undo capture (records into D4's log):** `apply_patch` records inverses into `rmc-host::undo::UndoLog` (P0.3-owned) — for each key it will `put`/`delete` it captures the prior `get`/duplicate-presence as an inverse entry. This step only RECORDS; the replay/`invert` that rolls the working snapshot back without a fresh `mdb_copy` lives with P0.3/D4.
- **Determinism cross-check (P0.1):** an incremental `apply_patch` from base→edit must leave the working env byte-equivalent (on reward-bearing tables) to a cold `write_model` of the same post-edit tree — the differential test that guards every patch.

**Key logic / algorithm:**
- **Undo journal feeds off the apply path (D4 "per-op undo log, not full LMDB copies").** Before each LMDB mutation in P0.2's `apply_patch`/`compute_patch` (which `put`s into the same tables `write_model` does — `nodes_by_id`, `bindings_by_*`, `children_by_parent`, `usages_by_*`, `signatures_by_target`, `static_metadata_by_target`, `meta_by_key`), call `capture_prior` and push an `UndoEntry`. `checkpoint()` just records `undo.mark()` — O(1), no copy. Embeddings/descriptions caches are content-hash keyed (D3 `C`), so they are *not* journaled — they self-heal on next read.
- **`restore_atomic` resets the four layers in dependency order, all-or-nothing:** (1) `vcs::op_restore(cp.source)` rewinds source files to the checkpoint jj operation; (2) open one working-snapshot `wtxn`, `undo.replay_to(cp.graph)` inverts every LMDB patch back to the marker, then `wtxn.commit()` (single transaction = graph atomicity for free, mirroring `write_model`'s one-txn commit); (3) re-sync the warm RA host by replaying inverse `set_file_text` to `cp.ra_edit_seq` over the same dirty files (reuses P0.2's `set_file_text` ingestion); (4) caches need no action. Then `undo.truncate_to(cp.graph)`.
- **Atomicity escape hatch (D4 "a failed restore re-opens the working snapshot from base").** If undo replay or host re-sync fails, abort the partial reset and re-init the working snapshot from the base published snapshot via the P0.2 `mdb_copy`-from-base path + re-open RA host from base, then surface `RestoreError::FellBackToBase`. Slow path, explicitly avoided in the happy case.
- **jj wrapper stays thin and subprocess-based** (jj 0.40.0 confirmed present; `jj op log`/`jj op restore` are the primitives). It only knows source; the contract's value is that P0.3 binds it to graph+host+caches so `rollback()` is never "just a jj wrapper" (the design's explicit warning).
- **Working-snapshot identity is respected (D1).** Restore decrements the host's `edit_seq` back to `cp.ra_edit_seq`; identity `(session_id, base_graph_id, edit_seq)` is content-decoupled, so reversing patches in place is legal — no new content-addressed `graph_id` is minted (unlike published snapshots in `storage::graph_id_for`).

**Depends on:** [P0.2].
**LOC:** prod +408 / test +490.

**Risks / unknowns:**
- jj op-id template: `jj op log -T 'id.short()'` flag/format must be verified against jj 0.40.0 (verify) — fallback is parsing the leading hex token of default `jj op log --no-graph` output (the format observed in this repo).
- RA host inverse-`set_file_text` replay may not reproduce the exact pre-edit salsa state if intermediate edits touched the same file non-monotonically; the D4 fallback (re-open from base) covers correctness but is slow — needs the P0.2 fan-out measurement to know how often it triggers.
- DUP_SORT inverse precision: deleting the *one* `(key,val)` pair we added (not the whole duplicate set) requires heed `delete_one`/cursor semantics; a wrong delete silently corrupts secondaries → corrupts reward (mirrors implementation issue #5). Differential test (restore == cold-rebuild-of-pre-state) on every op is the guard.
- `jj op restore` rewinds the *whole working copy*; if the episode is the only writer this is fine, but concurrent rollout workspaces must each be a distinct jj workspace/op-head (verify P0.4 pool gives one jj repo per workspace clone).
- Counter inverses (`meta_by_key` usage/node counts, D3 `P`) must be journaled too, else manifest/`meta` counts drift after rollback — easy to miss; covered by `MetaCounter` variant.
- `Instant` in `Checkpoint` is non-serializable; if checkpoints must persist across a process restart for crash-recovery, swap to a logical seq (out of scope here — checkpoints are in-session/ephemeral like the working snapshot).

**Key logic / algorithm:**
- **Reuse the embedding cache, do not re-embed.** `build_feature_matrix` calls `rmc_graph::graph::embedding_cache::prepare_embeddings_for` + `embed_prepared_embeddings` (or `ensure_embeddings_for`) which already hit `embeddings_by_target` by `content_hash` + `embedder_version`; pack the returned `ResolvedEmbedding.vector`s row-major into a Candle `Tensor` (`Device::cuda_if_available`, falling back to CPU). Dim from `backend.dim()` (1024 default).
- **Item selection is shared with `semantic_overlaps`.** Reuse `enumerate_similarity_seeds` (skips `NodeKind != Item`, `file/span == None`, `::tests::`) so the `Analyze` view sees the same population the existing similarity tool does — no second definition of "what is an item."
- **Adjacency is consumed, not rebuilt — consumes `rmc-graph` `call_adjacency` (D).** No private `structural_adjacency` here; `build_feature_matrix` fetches the shared call/usage adjacency from `rmc-graph`'s `call_adjacency` (and `crate_edge_graph` for crate-level structure), reindexed onto the item set, as the spectral input. The shared builder already does a single-scan walk, so there are no per-pair queries.
- **`Substrate::Embedding`** runs GMM (soft membership → overlapping concepts, e.g. "parse AND cache"); **`Substrate::Structure`** runs spectral on the `rmc-graph` `call_adjacency` (Laplacian fuses graph topology); **`Substrate::Both`** concatenates L2-normalized embedding rows with normalized spectral coordinates before GMM. GMM/spectral both bottom out in `kmeans_pp`.
- **Seeded determinism (right-sized rigor).** One `AnalyzeSeed` per run; `rng_for("kmeans-init")`, `rng_for("gmm-em")` derive independent `ChaCha8Rng` streams so cluster *labels* are reproducible run-to-run. Per plan §"Right-Sized Rigor": no deterministic GPU kernels chased — only the seed is fixed and logged. Cosine geometry stays consistent with `graph::math::cosine` for centroid scoring.
- **is/ought discipline.** Output is pure `Cluster` membership + `crate_spread` (a *fact*: "this concept spans N crates"); the engine attaches no refactor suggestion. `ClusterId` is run-local and not persisted/content-addressed — clusters are perception, regenerated once per episode and cached by the caller (P1.3 step owner), not stored in LMDB.
- **Differential safety net.** Keep `union_find_clusters` (the extracted `build_clusters`) as a cheap CPU fallback and a test oracle: on a tiny fixture the GMM/spectral connected-component structure must align with the threshold union-find result.

- **Adjacency cost is owned by `rmc-graph`.** Since the call/usage adjacency is consumed from `rmc-graph` `call_adjacency` (single-scan, `crate_edges`-style) rather than re-walked here, there is no per-pair `node_to_crate` cost in this crate; only the reindex onto the item set is local.

- **Outliers:** features are the `ItemFeatures` vectors built by P1.3a (embedding from `embedding_cache::ResolvedEmbedding.vector` concatenated with structural scalars: fan-in/out from `who_calls`/`calls_from` counts, body LOC, complexity). **Key logic / algorithm:**
- **move_item.** (1) classify the edit as `EditClass::ItemAddRemove` (D2) — a `pub` decl leaves one module and enters another, so the affected set is *editing crate (nodes/bindings/contains) + reverse-dep usages/bindings whose surface changed*; gather reverse-deps from the reversed `crate_edges()` consumer→producer graph plus `who_uses_targets([target])`. (2) `cut_decl_text` reads `Node.file`+`Node.span` from the working snapshot; `insert_decl_and_use` writes the decl into the dest module's file and inserts a `use <new_path>;` in the old module so unqualified local references still resolve. (3) Call `WorkspaceHost::apply_edits` (P0.2) with the dirty files; then `move_decl_references` runs RA's `analysis.rename`/`SourceChange` path (rename.rs) — repurposed not to rename but to recompute and rewrite fully-qualified `use`/path references across reverse-dep files, applied via the new `apply_source_change`. (4) Re-extract the affected set via `emit_crate_scoped` and `compute_patch`/`apply_patch` (P0.2 / rmc-graph). (5) Gate: `assert_no_new_cycle` calls `OpenedSnapshot::would_create_crate_cycle(consumer, producer)` (consumes the canonical crate-cycle detector from rmc-graph) — if it returns `true` (the dest crate would now depend back on a crate that depends on it), refuse with a `RefusalReason::Cycle{principle:"P3"}` and roll back the working-snapshot patch via the D4 undo log.
- **delete.** (1) `who_uses_targets([target])` (plus `who_imports`) collects all referencing consumers. (2) If non-empty and `cascade == false` ⇒ `DeleteError::StillReferenced` → structured `Refusal` (suggested_actions: `inline`, `move_target`, `delete_dependents_first`). (3) If `cascade`, transitively collect dependents (bounded reverse-dep walk over `who_uses_targets`), delete each decl + strip now-dangling `use` lines (reuse `imports_of` to find them), classify as `ItemAddRemove`, build the union affected set, patch LMDB.
- **Cycle check substrate.** The crate-cycle gate consumes `OpenedSnapshot::would_create_crate_cycle(consumer, producer) -> bool` from rmc-graph (no private `crate_edges_scc`/DFS is defined here, and no new workspace dep is introduced): given the proposed dest crate (`consumer`) and the crate that owns the moved decl's current dependents (`producer`), the method reports whether adding that consumer→producer edge closes a crate-level cycle. This is the canonical P3 detector.
- **Propagation reuse.** RA rename today is *preview-only* (`source_change_to_preview` in rename.rs builds a `RenamePreview` but never writes); this step owns and adds the apply half (`apply_source_change`, pinned `pub fn apply_source_change(host: &mut WorkspaceHost, change: SourceChange) -> Result<Vec<PathBuf>, ApplyError>`) and is the first consumer of it — `modify_signature` (P1.5c) reuses the same applied-`SourceChange` plumbing.
- **Differential safety.** Per cross-cutting P1.5 issue #3, the integration test runs each op then re-extracts cold (`graph::extract::extract`) and asserts the patched working snapshot's node/binding/usage sets equal the cold rebuild's.

**Depends on:** [P1.5a, P0.2, P0.3].

**LOC:** prod +780 / test +395.

**Risks / unknowns:**
- RA `rename` was built to rename a *symbol*, not to recompute import paths after a *move*; using `SourceChange` to fix `use` paths may require synthesizing a small AST edit instead, or a path-rewrite pass independent of `analysis.rename` (verify the RA API supports "item moved to new module, fix imports" — it may not, in which case `move_decl_references` becomes a hand-rolled `use`-path rewriter over `imports_of`/`usages_of`). This is the biggest unknown in the step.
- `apply_source_change` is genuinely new behavior on a preview-only surface; RA can refuse (keywords, foreign items, conflicting edits) — must surface those as `MoveError::RaRefused` not a panic.
- Cascade-delete blast radius: a transitive reverse-dep walk can touch many crates; bound it and prefer refuse. Default is refuse (open decision #2 in v3 leans `(a) refuse`).
- The `would_create_crate_cycle` gate only catches *crate-level* cycles; an intra-crate module cycle from a move is not a P3 crate cycle but may still be undesirable — out of scope here (verify P3 scope is crate-level only).
- `Node.span` is `Option<(u32, u32)>` (byte offsets, verify) — decl-text extraction must handle attributes/doc-comments preceding the item (the `attributes` Vec is separate from the span) so `cut_decl_text` doesn't orphan `#[...]` lines.
- Reverse-dep affected-set correctness is load-bearing for training signal; covered by the differential test but the union-of-affected-sets logic for cascade is fiddly.

- **Two mechanisms, one op.** RA's `analysis.rename` (in `rename_at_file_position`) only substitutes an *identifier* at decl + every ref — it cannot add/remove/reorder params. So `classify` splits: `NameOnly` ⇒ reuse the RA rename `SourceChange` (now *applied*, not previewed); `Surgery` ⇒ (a) rewrite the decl span from `format_signature(new_sig)`, (b) for added params synthesize args at each call site, (c) for removed params delete the arg at each call site. Call sites come from `OpenedSnapshot::function_signature`'s sibling data: the `usages_by_target` records (`Usage{file,start,end,consumer_function}`) plus RA `find_all_refs` (position.rs) for cross-crate refs the affected-set (D2 Sig/vis) covers.
- **The biggest correctness call (callsite synthesis for added params).** Default `Refuse`: an added param the engine cannot fill ⇒ `AddedParamUnsynthesizable` hard refusal with a `suggested_actions` reason (the "restructure, don't suppress" discipline). `TodoStub`: insert `todo!()` and attach a required follow-up so the trajectory must resolve it before `declare_done`. `Strategy`: agent supplies one `ArgExpr` per added param, inserted verbatim — no inference, the architectural judgment stays in the policy. Removed/reordered params are mechanical (delete/permute the arg list) and never refuse.
- **Promote preview→apply.** Today `source_change_to_preview` only maps `SourceChange.source_file_edits`/`file_system_edits` into a `RenamePreview`; `rename_at_file_position` returns it without touching disk. The host `ApplyError` (chain `PatchError -> rmc_host::ApplyError -> CrudError`) surfaces as `ModifySignatureError::Apply`.
- **Edit class = Sig/vis (D2/D3).** A signature change re-extracts the editing crate's `nodes_by_id`+`signatures_by_target` and reverse-deps that reference the item (their `usages_by_id`+secondaries re-resolve). `format_signature` feeds the decl rewrite; `compute_patch`/`apply_patch` (rmc-graph) produce the `GraphPatch`; the prior values become the `UndoEntry` for the D4 `Checkpoint`.
- **RA refusals are first-class.** `analysis.rename` already returns `Err` for keywords/foreign items/conflicts (today mapped to `anyhow`); P1.5c surfaces that as `RenameRefused{ra_msg}` so the gate layer can convert it to a structured `Refusal` the policy learns from, not a crash.

- **Span→FilePosition fidelity.** `Node.span` is `(u32,u32)` byte offsets (verify these are file-byte offsets, not within-item) — need a clean byte-offset→`FilePosition` path; `RenameEdit` is line/col while the host wants offsets, so a consistent coordinate space across rename.rs and the host is load-bearing (mismatch silently corrupts edits → corrupt reward, Issue #3).

**Key logic / algorithm:**
- All six verbs lower to a single `ModuleEditPlan` (file writes/deletes + `mod`-decl edits + optional `ManifestEdit`s) handed to `WorkspaceHost::apply_edits` (rmc-host), so they ride the same apply==rebuild engine as P1.5a–d; only the `EditClass` differs.

- Split/merge validation is pure *is/ought*: the agent supplies the `Partition`; the engine only checks total coverage (every declared child item assigned exactly once) and name collisions — it never proposes a partition.
- Crate ops do `Cargo.toml` surgery with `toml_edit` (preserve formatting): `lift_to_crate` appends to root `[workspace] members` and adds a `path` dep on the consumer; identity flows through `loader::load_crate_target_kinds` (`crate_target_kinds_by_name`) and `emit_crate`'s `crate_id = NodeId::from_components(["crate", crate_name])`, so a new crate name = a new crate node. CargoManifest is the D2 cold-rebuild class (feature/member unification can ripple workspace-wide) → after a crate op the host re-`load()`s instead of incremental patch. (`toml_edit` is inherited from the workspace dep declared by P1.X via `workspace = true` — this section adds no new `[workspace.dependencies]` entry.)
- Cycle safety for `lift_to_crate`/`move_module`: before writing, call the shared rmc-graph `would_create_crate_cycle` (no private cycle impl in rmc-env) — it returns `bool`; on `true` the op constructs `Refusal { reason: RefusalReason::Cycle, .. }` (P1.6 canonical; `RefusalReason::Cycle` owned by P1.6) and applies nothing. The cycle check itself is a pure predicate, not an error path — `ModuleOpError` carries no `Cycle` variant.
- Every op is wrapped in a D4 `Checkpoint` (rmc-host `UndoLog` marker + jj op id + RA edit-seq); the integration test diffs incremental-apply graph against a cold rebuild of the same final tree to catch silent ModuleTree-patch corruption.

**Depends on:** [P0.2 (WorkspaceHost/EditClass/AffectedSet/apply_edits), P0.3 (Checkpoint/rollback), P1.5a (CrudEngine + apply==rebuild), P1.5b (use-path rewrite + reverse-dep affected sets), P1.6 (GuidelineGate/Refusal for cycle refusals), rmc-graph emit_crate_scoped].

**Risks / unknowns:**
- File-based-module surgery is multi-file and ordering-sensitive (create child file before inserting its `mod` decl, remove decl before deleting file); a half-applied plan must roll back cleanly through the D4 undo log — transactionality across N file writes is the main correctness hazard.
- `move_module` / `lower_to_module` cross-crate path rewrites (`crate::x` ↔ `othercrate::x`) reuse RA-assisted rename, which is **preview-only today** (per implementation plan) — applying and handling RA refusals (keyword/foreign-item conflicts) is unproven for module-granularity moves. (verify)

**Key logic / algorithm:**
- **Single source of truth.** P1.4 introduces no apply split of its own. The `compute_effects` (pure) + `persist` (effectful) factoring of `CrudEngine::apply` is **owned by P1.5a** (engine file `crud.rs`); P1.4 *consumes* `Effects` and `compute_effects` from P1.5a. The simulator is a **dry-run**: `simulate == predict∘compute_effects` and it SKIPS `persist` entirely. Because `simulate` and `apply` funnel through the identical P1.5a `compute_effects`, the simulator's `would_refuse` and `edit_class`/`affected`/`patch` are byte-identical to what apply would do — it structurally cannot lie (resolves issue #7).
- **Sim types are shared.** `AuditDelta` and `GraphMetricsDelta` are *imported from* `rmc-env::reward_model` (C); P1.4 defines no local `AuditDelta` and no local `MetricDelta` (the old metric-delta is merged into `GraphMetricsDelta`). This keeps the simulator's deltas identical to the reward vector the policy trains on.
- **Dry-run via uncommitted txn, not a copy.** `predict::from_effects` does **not** `mdb_copy` the working snapshot. It opens one `RwTxn`, applies `eff.patch` (reuse `apply_patch`), runs the affected-set-scoped audits/metrics through the existing read API, then drops the txn so LMDB discards the writes. This keeps `simulate` cheap enough to call as a planning primitive (the design's "cheap lookahead").
- **Audit deltas via the shared battery.** `audit_delta` is computed by calling `rmc-env::audit_battery::run_audit_battery` once on the baseline and once on the overlay (scoped to `AffectedSet`) and subtracting — P1.4 does **not** define a private `predict_audit_delta` orchestration; it reuses the same battery the reward path (P1.7) and gate path (P1.6) drive. `metric_delta` reuses `OpenedSnapshot::crate_dependency_metric` + the petgraph metrics from P1.7.
- **Cascade = the D2 affected-set, rendered.** `Cascade` is just `eff.affected` (crates + reverse-dep item count from `affected_set()` over reversed `crate_edges`) plus the touched callers from `who_calls`/`usages_by_target`, formatted as `Location`s for the agent's view — the same set `persist` would re-extract.
- **Cycle/adjacency via shared graph builders.** The cycle/import-cycle refusal probe uses the rmc-graph shared builders `would_create_crate_cycle` and `crate_edge_graph` over the overlay — no private DFS or adjacency implementation lives in P1.4.
- **No source write on the simulate path.** `compute_effects` (P1.5a) builds `SourceEdit`s in memory and (for refusal gates that need typed HIR, e.g. cycle/visibility checks) uses the warm host's already-loaded `RootDatabase` via a throwaway `set_file_text`+rollback only if a gate truly needs post-edit semantics; default gates run on the `GraphPatch` overlay to avoid touching the host at all.

**Depends on:** [P1.5a, P0.2c, P1.6, P1.7].

**LOC:** prod +471 / test +365.

**Risks / unknowns:**
- **Overlay-txn read API.** `query/*` methods take `&OpenedSnapshot` bound to an `Env`; exposing a read view over an open-but-uncommitted `RwTxn` may require a small lifetime/trait adapter on `OpenedSnapshot` (verify heed allows reads through the same `RwTxn`). If not, fall back to a scratch sub-env populated from the patch.
- **Gate-dry-run coupling.** Some P1.6 hard refusals (cycle introduction, trait coherence) need *post-edit* RA semantics, not just the graph patch — those force a warm-host `set_file_text`+rollback inside the P1.5a `compute_effects`, making `simulate` no longer free of host mutation. Need to confirm which gates are graph-only vs HIR-requiring; HIR-requiring ones make repeated `simulate` calls as costly as a partial apply.
- **Delta baseline drift.** `predict` diffs against the *current* working snapshot; if the agent simulates several ops in a row, each is independent (no stacking). Stacked/what-if-chains are out of scope here.
- **`compute_effects` purity is load-bearing and untested by the type system.** A future contributor adding an IO call inside P1.5a's `compute_effects` silently re-introduces the lie. Mitigate with the differential test (`tests/simulate_matches_apply.rs`) run in CI on every edit class, and a doc-comment contract; consider a `#[must_use]` `Effects` to discourage discarding without `persist`.
- **petgraph metric cost.** modularity/conductance on the affected subgraph is fine; a full-workspace recompute per `simulate` would re-create the latency wall — must scope to `AffectedSet` (shared concern with P1.7).

**Key logic / algorithm:**
- The harness is a *post-apply* check: `CrudEngine` (P1.5a) calls `GuidelineGate::check` after `WorkspaceHost::apply` has patched the working snapshot, passing the op's `AffectedSet` (P0.2/D2). From the affected set we take the editing crate's `NodeId` as `crate_id_filter` and the dirty `FileId`s to build `only_files`, so the audit cost is proportional to the edit, not the workspace (resolves Issue: "audit latency at write time").
- Pattern and `unsafe` violations are NOT orchestrated locally: the gate consumes the audit set from `rmc-env::audit_battery::run_audit_battery`, which runs the eight built-in matchers (`match_unwrap`, `match_expect`, `match_panic_macros`, `match_unwrap_unchecked`, `match_unbounded_loop`, `match_await_in_guard_scope`, `match_transmute`, `match_self_recursion`) dirty-scoped via `fn_body_audit::fn_body_audit_scoped` plus `graph::unsafe_audit::unsafe_audit_impl` (one `UnsafeFinding` per `unsafe {}`). Each returned `FnBodyFinding.pattern` string maps through `classify()` to `Hard`/`Soft`; `unsafe` findings are gated only when *not* `has_safety_comment` and *not* allowlisted.
- Numeric body guidelines (cyclomatic / param count / body LOC / nesting) are NOT part of the shared battery — the gate layers them ON TOP via `compute_body_metrics`. It reads `OpenedSnapshot::function_signature(target).params.len()` for the param count (no AST needed) and walks the `ast::Fn` body the same way `fn_body_audit` already parses files (`sema.parse_guess_edition` → `descendants` → `ast::Fn::cast` → `body()`). The decision-point count is the AST equivalent of the regex heuristic in `analyze_complexity` (analysis.rs:477-487) but node-based, not line-based, so it is robust to formatting.
- The gate IMPORTS `Severity`, `GateOutcome`, and `PrincipleViolation` from `rmc-env::reward_model` (the canonical owner) rather than defining them: `consumes Severity/GateOutcome/PrincipleViolation from rmc-env::reward_model`. The old local `Penalty` type is gone — it is merged into `PrincipleViolation` (`principle`/`severity`/`weight` + optional `detail`), and the gate emits a `Vec<PrincipleViolation>` for the soft path.
- The cycle (P3) and visibility-widening (P13) refusal *reasons* live in `RefusalReason` here, but are *produced* by the CRUD ops: P1.5b's `cycle_gate::assert_no_new_cycle` (via `crate_edges()` SCC) constructs `RefusalReason::Cycle`, and a future visibility op consults `dead_pub_in_crate`/`BindingVisibility`. The gate centralizes only the type + JSON shape + severity so every refusal is uniform training data.
- The `BoundaryAllowlist` is loaded once from a workspace TOML (`[[boundary]] target/reason`, per design), held immutable in `GuidelineGate`; `excuses()` is consulted before promoting an `Unsafe`/`Transmute` finding to a `Refusal` (the only sanctioned bypass). No agent-facing mutation method is compiled, satisfying governing principle #2 ("the agent never bypasses guidelines").
- `GateOutcome::Refused` signals `CrudEngine` to roll back the working-snapshot patch via the D4 undo marker (no source left mutated); `Pass{violations}` flows the soft `PrincipleViolation` list into P1.7's `RewardVector.principle_violations`. Compile/test gates (`CompileBreak`/`TestFailure`) are *reserved* variants here but actually evaluated in P1.7 (cargo/RA gate) — P1.6 owns only the static, dirty-scoped audits.

- Threshold calibration is unspecified by the design — the `GateConfig` defaults (max_cyclomatic/params/LOC/nesting) are guesses (verify) and likely need tuning against the rmc repo's own functions so baseline code is not all-refused.

(accept; document.)

**Key logic / algorithm:**
- **Shared reward vocabulary lives in `reward_model`.** `Severity`/`GateOutcome`/`PrincipleViolation`/`AuditSnapshot`/`AuditDelta`/`GraphMetrics`/`GraphMetricsDelta` are defined once in `crates/rmc-env/src/reward_model.rs` (this section owns them) and imported by `reward.rs`, the `audit_delta`/`graph_metrics` submodules, P1.6, and P1.4 — so the gate, the planner, and the reward producer all speak the same types with no duplicate defs to drift.
- **Audit orchestration is one battery.** `audit_battery::run_audit_battery(loaded, snap, scope)` is the single place the audit calls are listed; `capture_audit_snapshot` just delegates to it. The battery calls the existing audits unchanged: `recursion_check(snap, opts)` for cycle count, `snap.dead_pub_report()` summed for dead-pub, `unsafe_audit(...)` filtered against the boundary allowlist (P1.6), `fn_body_audit(loaded, snap, opts)` for body-violation count, and `snap.crate_dependency_metric()` → mean `instability`/`abstractness` (the `CrateMetric` fields). `diff_audit` subtracts. Per the is/ought discipline, reward is keyed on these *deltas improving*, never on "acted on a flagged item." Per-step scope is the dirty crate only (fast); `declare_done` is workspace-wide.
- **Two-tier gate (M0 Spike 2 fallback, the likely branch).** Per-step CRUD `commit` runs `RaGate::diagnostics` over the dirty `FileId`s on the already-warm host — near-free, type-level only. The real correctness gate (`CargoGate::check` + scoped `cargo test`) runs only at `declare_done` (P1.8), via `nix develop ../nix-devshells#cuda-code --command cargo …` per AGENTS.md. `GateTier::{PerStep, EpisodeEnd}` selects which runs. Borrowck/trait errors only surface at episode end — accepted and shaped for, exactly as Spike 2 prescribes; if Spike 2 reports `t_warm_check < ~2s` on T2 the per-step tier swaps to cargo-check with no type changes.
- **Graph metrics over petgraph (new dep).** `capture_graph_metrics` builds a `petgraph::Graph` from the shared `crate_edge_graph()` (crate-level) and `module_edge_pairs()` (module-level) rmc-graph builders — combinatorial, CPU, O(V+E), cheap at workspace scale (design: petgraph for graph algos, Candle stays for embeddings). Compute modularity (resolution-corrected Newman), conductance (boundary/volume), clustering coefficient (transitivity), Brandes betweenness-max. These score state *after* the move and feed `graph_metrics_delta`; they are reward shaping, not an objective the agent reads (Goodhart hardening deferred until gaming is observed).
- **Hard-fail → D4 rollback.** On `GateFailure` (compile break, scoped-test failure, or an escalated hard `Refusal`), `commit` calls `host.restore(cp)` (P0.3's atomic four-layer reset to the pre-op `Checkpoint`) so no partial source/graph/host state survives, and returns `CommitOutcome::HardFailed { restored_to, failure }`. The per-op `Checkpoint` is taken by the CRUD verb (P1.5a) before apply; `commit` only needs the handle.
- **Delta computation is incremental-friendly.** The pre-op `AuditSnapshot`/`GraphMetrics` are captured once at episode start (or carried from the prior step's after-state), so `commit` re-derives only the after-state over the dirty scope — avoiding the full-recompute latency wall the design warns about for per-commit metric deltas.

**Risks / unknowns:**
- **Cargo gate latency (Issue #2, second latency monster).** `cargo check`/`test` per commit is seconds-to-minutes; the whole design here assumes Spike 2's two-tier fallback (RA per step, cargo at `declare_done`). If even episode-end cargo is too slow on T2, commit batching or mid-episode error tolerance must be revisited. The `nix develop` wrapper adds shell spin-up overhead on every invocation — measure whether to keep a long-lived devshell process.
- **Graph-metric delta must stay incremental.** A naive full `crate_edge_graph()` rebuild + Brandes betweenness per commit re-creates the latency wall; betweenness is O(V·E) and is the costliest term — may need to cap it to crate-scale only (small V) or recompute it lazily (per-episode, not per-step) while modularity/conductance run per commit.
- **Scalarization weights are guesswork (verify).** `RewardConfig` is fixed per phase but the relative weights (gate vs audit-delta vs metric-delta vs length) are unknown until P1.8 rollouts show what dominates the gradient; credit assignment (commit-time compile break vs the 5 prior moves) is P1.8's problem but the reward shape constrains it.
- **RA diagnostics coverage (M0 Spike 2 coverage probe).** RA catches type-mismatch/missing-field/unresolved but misses borrowck and may miss trait-bound errors — the per-step gate is a type-level filter only, not a soundness guarantee. **Key logic / algorithm:**
- **The action layer is consumed, not defined.** P1.8 consumes `Action` and its five sub-enums (`Navigate`/`Analyze`/`Query`/`Crud`/`Meta`), `ActionCategory`, `Observation`, and `EnvSession` from P1.X (in `tools/env/`). The whole `Action` enum is `Serialize`/`Deserialize` (per P1.X) so it doubles as the policy's action schema and the trajectory's logged action with no separate DTO. The runner never decomposes a labeled intent; the policy emits primitives and the runner replays them through `EnvSession`/`Action::dispatch`.
- **The loop is the design's Episode Shape, verbatim.** `run` mirrors the v3 pseudocode: `obs = build_observation` (P1.1 `ContextView` + cached P1.3 `VisionMap`/`Outlier` + P1.2 `Description`); `action = policy.choose(obs, available)`; the runner hands `action` to P1.X's `EnvSession`/`Action::dispatch`, which yields an `Observation`: navigate/analyze/query → observed + budget debit (the observation cost = "budget pressure"); Crud → `CrudEngine` returns `CrudOutcome::{Applied,Refused}` (P1.5a `modify_body` is the only wired verb at M3; b–e widen it); `Meta::Commit` → P1.7 full `RewardVector`; `Meta::DeclareDone`/budget-exhaustion → break with `EpisodeOutcome`. Most trajectories naturally become "attempt body → blocked by gate → must restructure," the trained skill.
- **`run_episode` is a thin server-side driver (consumes `EnvSession` + `EnvStepParams` from P1.X).** The MCP entrypoint does not own the action layer or define its own params: it deserializes P1.X's `EnvStepParams`, opens the canonical `EnvSession` via the `env_step` session API, and drives `EpisodeRunner::run` over it. It lives in `tools/env/episode.rs`, not a separate `endpoints/env.rs`.
- **Borrow constraint (§J) — `EnvSession` owns `WorkspaceHost` by value; `dispatch` reborrows per-arm.** P1.X's `EnvSession` holds the `WorkspaceHost` by value; `Action::dispatch` reborrows the host inside each match arm and constructs the per-arm engine on demand (`CrudEngine` with `&mut WorkspaceHost`, `Simulator` with `&WorkspaceHost`) — engines are built per-dispatch, never stored holding a borrow across steps. The runner therefore never threads two live host borrows through one struct; it calls `dispatch` once per step and lets each arm reborrow. P1.8 honors this by keeping only the `Policy`/`TrajectoryWriter`/cumulative-reward on the runner and leaving all host + world state (and the per-arm reborrow) to `EnvSession`.
- **Per-episode-once vs per-step (right-sized rigor / latency).** `EpisodeRunner::run` computes the Analyze layer ONCE at episode start (`ClusterEngine` + `outliers` + `affinity` + `VisionMap`, P1.3) and the description index once, caching both on `EnvSession`; the per-step cost is only the policy forward pass + the cheap incremental view assembly + (on CRUD) the P0.2 sub-second re-extract. This honors the "run expensive global stuff less often than cheap local stuff" rule without any enforcement machinery.
- **Every step is checkpointed; the trajectory is the audit log.** Before each CRUD/commit the runner takes a D4 `Checkpoint` (P0.3) and records its `CheckpointId` into the `TrajectoryStep`, so `Meta::Rollback` is `WorkspaceHost::rollback` to a prior step and "how did this reach this state?" is answerable from the trajectory alone. A hard-fail commit auto-rolls-back (P1.7 contract). Refusals are first-class `TrajectoryStep`s (`StepRecord::Refused`) — every refusal is training data per the design.
- **SFT data format is locked by `trajectory_format.rs`.** `Trajectory` serializes to JSONL: one `TrajectoryStep` per line (streamed via `TrajectoryWriter::append` so partial runs survive a crash) + a trailing `EpisodeOutcome`. Each step carries the full `ContextView` (the prompt the policy saw), the `available: Vec<ActionCategory>` (the legal-move mask), the chosen `Action`, and the `StepRecord` (reward / refusal) — exactly the (observation, action, reward) tuple SFT/RL consume. The golden test pins the schema so downstream training code does not break on a field rename.
- **Frontier-model harness.** `ApiPolicy` renders the structured-text view (P1.1 `render_context_view`) + the legal action list into a single chat turn, advertises the `Action` JSON/tool schema (P1.X), and parses the reply back into an `Action`; `EpisodeRunner` is policy-agnostic (`dyn Policy`), so the same loop runs the `ScriptedPolicy` in CI (deterministic, no network — the M3 `episode_modify_body_e2e` test) and a real model via the `run_episode` MCP tool on the rmc repo's own `project_paths` dedupe (the M3 exit criterion).

**Risks / unknowns:**
- **Credit assignment (Issue, design + register).** A commit-time compile/test break is attributable to the commit step but caused by ≤N prior CRUD moves; the trajectory logs per-step rewards + the episode summary, but the *scalarization across steps* (how the gate failure back-propagates) is a training-side decision deliberately left to Phase 3 — P1.8 only guarantees the data is captured, not the credit rule.
- **`ContextView` is heavy to log every step.** Embedding the full view per `TrajectoryStep` makes JSONL large; may need a `view_digest` (hash + delta-from-previous) instead of the full view for long episodes — decide before freezing the schema (the golden test would have to bless either form). (verify acceptable size on a 50-step episode.)
- **Expressiveness / abstraction leak (register Issue).** The five-verb set (owned by P1.X) may not cover every refactor the rmc `project_paths` task needs (e.g. a cross-crate helper extraction that doesn't fit `extract_function`/`lift_to_crate`); the M3 run is also the test of whether the action space is complete — a stuck frontier model is a signal the verb set needs widening, not a runner bug.
- **`ApiPolicy` action parsing robustness.** Frontier models emit malformed/partial tool calls; `Unparseable` + single retry is the floor, but a model that never produces a valid `Action` stalls the episode — needs a max-reparse-attempt cap folded into `StepBudget` (else an episode burns the whole budget on parse failures). (verify the chat/tool-call schema matches the chosen provider; reuses the `embeddings/openrouter/client.rs` reqwest precedent, but that is /embeddings not /chat.)
- **`run_episode` MCP tool is long-running** (seconds-to-minutes per episode under the P1.7 cargo gate) — the existing `#[tool]` methods are request/response; an episode may exceed MCP client timeouts. May need to return a job handle + poll (like the background sync task in `ServerRuntime`) rather than block, or run episodes only via a dedicated `rmc-env` binary/`rmc-bench` driver and keep the MCP tool for short scripted demos. (verify MCP timeout behavior; the table assumes the simple blocking form.)
- **Concurrent rollouts multiply the warm `WorkspaceHost`** (memory, register Issue #4): one `EnvSession` per concurrent episode holds a full RA `RootDatabase`. P1.8 runs single-episode; the mpsc+oneshot channel-actor wrapper for the host (deferred in P0.2a) lands here only if parallel rollouts are needed, and is noted as the scaling path, not built now.
- **§J borrow constraint may still reshape per-arm engine construction.** P1.X's `EnvSession` owns `WorkspaceHost` by value and `dispatch` reborrows per-arm (`CrudEngine` `&mut`, `Simulator` `&`), building engines per-dispatch rather than holding them across steps — the runner relies on engines being cheap to reconstruct each step. If an engine turns out to require long-lived state, the host must move behind the channel-actor; the per-dispatch reborrow assumption is the borrow-checker ergonomics risk to verify against the real P1.4/P1.5a signatures.

**Key logic / algorithm:**
- **Workspace wiring (dependency order matters).** Root `members` gains `rmc-host` → `rmc-describe` → `rmc-analyze` → `rmc-env` → `rmc-bench`; `default-members` stays `["crates/rust-code-mcp"]` so a plain `cargo build` does not pull the env/GPU stack. The five new third-party deps (`petgraph`, `rand`, `rand_chacha`, `toml_edit`, `fs_extra`) are declared once in `[workspace.dependencies]` and inherited — this is the single owner for them (the P1.3a/P1.3b/P1.5b/P1.5e/P0.4 sections each *assumed* they would add these; consolidate here to avoid duplicate/divergent version pins).
- **Top-level `Action` facade = the only new composition over the five steps.** Each verb category's sub-enum and its `apply` already live in its step (Navigate P1.1, Analyze P1.3, Query P1.2/P1.4, Crud P1.5, Meta P1.7/8). This step adds the *outer* `Action` enum + `dispatch`, so the env exposes one uniform `step(Action) -> Observation` to both the MCP boundary and the P1.8 episode runner. The Analyze/Query/Meta action *wrappers* are defined here because the P1.3/P1.4 sections deliver engine types but not the action enums (a real gap surfaced by the reconciliation).
- **MCP surface mirrors the existing pattern exactly.** New `#[tool]` methods on `SearchToolRouter` (router.rs) delegate to a `tools::env` family via `Parameters(...)` wrappers — identical shape to the 40+ existing tools (`build_hypergraph`, `semantic_overlaps`, …). The env state lives on `RuntimeState` as `EnvRegistry`, the same ownership model as `SemanticService`/`SearchRuntimeCache`, and is wired into `clear()`/`clear_runtime` so sessions are reclaimable. All of it is behind `#[cfg(feature = "env")]` (default off) so the existing read-only server is unchanged and the heavy CRUD/host stack is opt-in.
- **Config seam.** `EnvConfig` in rmc-config is the single place seeds/budget/reward-weights/allowlist-path are read, feeding `GlobalSeed` (P0.1), `AnalyzeSeed` (P1.3), `StepBudget`/`RewardConfig` (P1.7/8), and the boundary allowlist (P1.6). rmc-config currently has no `serde`/`toml` dep — added here.

**Risks / unknowns:**
- **`Action` as a single serde enum vs. the composition principle.** The design demands the policy emit *primitives*, not a labeled-intent enum; `Action` here is only the transport/dispatch union of the five primitive categories (not the demoted intent enum), but the boundary is subtle — `Action` must never grow a "do extract_shared_module" arm. Documented as a hard constraint; the enum-as-teacher scaffolding stays out of the deployed `Action`.
- **`EnvSession` is a large aggregate holding a warm `RootDatabase` + GPU embedding backend + LMDB working env per session.** One per concurrent rollout multiplies memory hard (Issues-register #4); `EnvRegistry` needs an eviction/cap policy that none of the underlying steps specify (verify — may need an LRU + explicit `env_close`).
- **Feature-flag matrix.** `rmc-env` transitively needs `rmc-analyze` (Candle/CUDA) and `rmc-describe` (reqwest LLM) and `rmc-host` (full RA stack); the `env` feature on rmc-server therefore pulls the entire heavy stack. Must confirm the CPU-only / no-CUDA build path compiles for tests (P1.3a flagged the same `Device::Cpu` uncertainty).
- **Two unresolved cross-step name conflicts (UndoLog dual definition #2, SCHEMA_VERSION double-bump #5) will break the build if both owners land independently** — these are the highest-priority reconciliation items and need an explicit owner assignment before P0.2c/P0.3/P1.2 are implemented.
- **rmcp `#[tool]` + a rich `Action` enum as a parameter:** `schemars::JsonSchema` must derive cleanly for the deeply-nested `Action` (it contains `Location`, `NodeId`, `Crud` payloads); if the derive is unwieldy over the full enum, `env_step` may need to accept a flattened/stringly action descriptor and parse server-side (verify schemars handles the nesting within the MCP token budget).
- **`fs_extra`/`toml_edit` version pins** are (verify) — confirm availability under the nix `cuda-code` devshell and that `toml_edit` coexists with the existing `toml = "0.9"` without a duplicate-crate conflict.
- **`default-members` discipline:** adding members must NOT change what `cargo build`/`cargo test` does by default for the existing binary; verify CI invocations target `-p` explicitly so the env/bench crates don't silently enter the default build/test set.
