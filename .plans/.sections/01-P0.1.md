**Key logic / algorithm:**
- IDs are already content-addressed and order-independent: `NodeId::from_components` (ids.rs) SHA-256s the component tuple, so the *set* of nodes/bindings/usages is reproducible for free. The defect is purely **iteration/insertion order** into the `Vec` fields of `ExtractionModel` (model.rs: `bindings`/`usages`/`contains`/`signatures`/`statics` are `Vec`; `nodes` is a `BTreeMap` and already fine).
- Primary nondeterminism source: `extract_usages` (usages.rs:45) iterates `for (&def_id, &target_node_id) in def_to_node` where `def_to_node: HashMap<ModuleDefId, NodeId>` — HashMap order is randomized per process, so `model.usages` (and `signatures`/`statics`, which also consume `def_to_node`) get nondeterministic push-order. Fix = iterate `def_to_node` in NodeId-byte order; belt-and-suspenders = `sort_canonical()` after extract.
- Secondary source: `extract_bindings` (bindings.rs:42) seeds from `module_node_for: HashMap`, and `emit_crate`/`extract_bindings` both walk `def_map.modules()` (RA-internal order, not guaranteed stable across loads). Final `sort_canonical()` normalizes the emitted `bindings`/`contains` regardless of walk order.
- `write_model` (snapshot.rs:407+) writes `nodes` from the BTreeMap and the Vecs in stored order, so once the model Vecs are canonicalized, `bindings_by_id`/`usages_by_id` primary keys (content-addressed via `binding_id_for`/`usage_id_for`) and the DUP_SORT secondaries (LMDB sorts duplicates internally) are reproducible.
- Seed: thread `GlobalSeed` through `BuildOptions` → `write_model` → `GraphManifest.seed` + `meta_by_key["seed"]`; `SCHEMA_VERSION` bumps 12→13 (graph_id_for hashes SCHEMA_VERSION, so old snapshots auto-rebuild — same pattern documented for v2..v12 in storage.rs).
- Golden test: `build_and_persist` twice into two `data_dir_override` temp dirs (force_rebuild), open both, assert `canonical_dump(a) == canonical_dump(b)`; a second in-process double-build catches HashMap-seed-dependent ordering that a single process might mask.

**Risks / unknowns:**
- `def_map.modules()` ordering across RA reloads is RA-internal; `sort_canonical()` covers `bindings`/`contains` but verify no other pass leaks RA iteration order into a persisted Vec (impls.rs `extract_impl_items` mutates `def_to_node` and `model.nodes`/`contains` — nodes via BTreeMap are safe, but its `contains`/binding pushes must also be caught by the final sort). (verify)
- `usages_by_id` keys are `usage_id_for(usage)` — confirm that function hashes only content fields (target/file/start/end/category), not insertion index, else the *key set* (not just order) drifts. (verify the exact fields of `usage_id_for` / `binding_id_for` in snapshot.rs — not yet read.)
- DUP_SORT value bytes are LMDB-sorted, so secondary-index order is already deterministic; the golden dump must compare them as sets, not insertion sequences, to avoid a false negative.
- SCHEMA_VERSION bump invalidates all existing on-disk snapshots (intended) — coordinate the single bump with any other M0 step that also bumps it to avoid a double increment.
- Right-sized-rigor scope: this guarantees cold-build reproducibility only. P0.2's warm-host incremental path must separately match these canonical orderings on the reward-bearing fields (cross-check via `canonical_dump` after each incremental patch); fully matching DUP_SORT ordering incrementally is out of scope here (issue #11 in the register).
