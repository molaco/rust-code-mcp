**Key logic / algorithm:**
- **Outliers:** features are the `ItemFeatures` vectors built by P1.3a (embedding from `embedding_cache::ResolvedEmbedding.vector` concatenated with structural scalars: fan-in/out from `who_calls`/`calls_from` counts, body LOC, complexity). LOF = k-NN local reachability-density ratio; Mahalanobis = centroid distance under regularized inverse covariance. Pure CPU (n≈workspace item count, O(n²) k-NN fine at this scale). Returns *facts* ("unusual on fan-in axis"), never "refactor."
- **Affinity:** build a `petgraph` adjacency once via `build_call_adjacency` (calling new `OpenedSnapshot::callees_adjacency` + `crate_edges`), run biased random walks → node2vec-lite structural vectors, combine `shared_callees` overlap, the walk-vector cosine, and the embedding cosine (`math::cosine`) into `Affinity.score`; explicitly carry `has_edge` so the agent sees latent pull between *unconnected* items — a feature, never a predicted edge to create (is/ought rule 2).
- **Co-change:** shell out to `git log --pretty=format:%H --name-only` at `OpenedSnapshot::workspace_root`, group changed files per commit, map files→`NodeId` via the node table's `file`+`span` (item granularity where spans allow, else file granularity), then Apriori frequent-pair mining with support/confidence/**lift** (P11 temporal coupling). Young repos → weak signal, surfaced via low `support` (recorded as Issue, not hidden).
- **Vision map:** `label_clusters` takes each `Cluster` centroid, queries `rmc-describe::DescriptionIndex` (P1.2) for the nearest description as the human-readable `ConceptLabel`, falling back to tf-idf over member display names; `build_vision_map` then renders clusters as `City` locations with inter-cluster `CityEdge`s (aggregated cross-cluster adjacency) and sets `fragmented_across_crates` when a city's members span ≥2 `Node.crate_id` — the exact "this concept is fragmented across crates" *is*-statement from the design view.
- **Determinism:** one `seed: u64` threaded into node2vec walks and any stochastic step; logged, not chased (right-sized rigor). All four run once per episode and are cached alongside P1.3a's clusters.

**Risks / unknowns:**
- **node2vec hyperparameters** (walk length, p/q, dim) are unspecified in the design — needs tuning; structural-feature quality is load-bearing for affinity perception (Issue #9 analog).
- **File→item mapping for co_change** is lossy: `Node.span` is `(start,end)` byte offsets but a commit diff is line-based; mapping a diff hunk to the enclosing item needs a line→item index, else co_change degrades to file granularity (verify whether file-level is acceptable for P11).
- **Git availability/perf:** `git log` over deep history can be slow on T2-scale repos; need a `--since`/`-n` cap in `CoChangeCfg`. Worktrees created by jj rollback (P0.3) may confuse `git log` path resolution (verify the working-snapshot dir is a real git checkout).
- **Mahalanobis on high-dim embeddings:** 1024-dim Qwen3 vectors → covariance is singular without dimensionality reduction; likely must run Mahalanobis on the PCA-reduced `ItemFeatures` P1.3a already computes, not raw embeddings (verify P1.3a exposes the reduced vectors).
- **`callees_adjacency` cost:** a full `usages_by_id` scan per episode is O(N_usages); acceptable once-per-episode but confirm it is not called per-step.
- **DescriptionIndex retrieval API** shape (P1.2) is assumed to expose nearest-description-by-vector; if it is text/BM25-only, `label_clusters` needs a different bridge (verify against P1.2's `DescriptionIndex`).
