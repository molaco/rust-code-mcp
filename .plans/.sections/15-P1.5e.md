**Functions / methods** (new/mod/del):
- `fn split_module(&mut self, op: ModuleOp) -> Result<AppliedEdit, ModuleOpError>` @ rmc-env::crud::module_ops — NEW — validate `Partition` covers exactly the module's declared items (else `PartitionNotCovering`/`OrphanItems`), create N submodule files, move item text + `mod` decls, leave a re-export façade in the parent (visibility preserved per Principle 13).
- `fn merge_modules(&mut self, op: ModuleOp) -> Result<AppliedEdit, ModuleOpError>` @ rmc-env::crud::module_ops — NEW — concatenate source module bodies into `dest`, drop the source `mod` decls + files, rewrite `use` paths that pointed at the old modules (RA-assisted, reuse the P1.5b path-fix machinery).
- `fn create_module(&mut self, op: ModuleOp) -> Result<AppliedEdit, ModuleOpError>` @ rmc-env::crud::module_ops — NEW — write an empty `name.rs` under the parent's dir and insert `mod name;` (file-based modules; never `mod.rs`).
- `fn move_module(&mut self, op: ModuleOp) -> Result<AppliedEdit, ModuleOpError>` @ rmc-env::crud::module_ops — NEW — relocate the module file/dir, remove the old `mod` decl, insert under the new parent, rewrite crate-internal `crate::old::path` references; refuse if it would create an intra-crate path cycle.
- `fn lift_to_crate(&mut self, op: CrateOp) -> Result<AppliedEdit, ModuleOpError>` @ rmc-env::crud::crate_ops — NEW — move a module subtree into a brand-new member crate dir (`new_crate/src/lib.rs` + sub-files), add the workspace member, add a `path = "..."` dependency from the origin crate, rewrite `crate::module::*` → `new_crate::*`; classify as **CargoManifest** → cold rebuild. **Pre-checks the new crate-dep edge for cycles against `crate_edges` (Principle 3) before writing.**
- `fn lower_to_module(&mut self, op: CrateOp) -> Result<AppliedEdit, ModuleOpError>` @ rmc-env::crud::crate_ops — NEW — inverse of lift: fold a member crate back into a module of the destination crate, remove the member + its `path` dep, rewrite `krate::*` → `crate::dest::*`.
- `fn plan_to_edit_class(plan: &ModuleEditPlan) -> EditClass` @ rmc-env::crud::mod_tree — NEW — `CargoManifest` if any `ManifestEdit` present, else `ModuleTree`.
- `fn locate_mod_decl(file: &str, name: &str) -> Option<TextRange>` @ rmc-env::crud::mod_tree — NEW — parse with `ra_ap_syntax` (already a dep) to find/insert/remove the `mod name;` item (no regex on Rust source).
- `fn apply_manifest_edits(root: &Path, edits: &[ManifestEdit]) -> Result<(), ManifestError>` @ rmc-env::crud::cargo_manifest — NEW — load `Cargo.toml` via `toml_edit`, apply members/dep deltas preserving formatting, write back.
- `fn emit_crate_scoped(model, db, vfs, root, krate, …) ` @ rmc-graph::graph::extract — NEW (extract from `emit_crate`) — re-run module/contains emission for a single crate after a `mod`-tree change; callable from `rmc-host` re-extract. Reuses `crate_def_map`, `module_path_segments`, `def_map.modules()`.

**Key logic / algorithm:**
- All six verbs lower to a single `ModuleEditPlan` (file writes/deletes + `mod`-decl edits + optional `ManifestEdit`s) handed to `WorkspaceHost::apply_edits` (rmc-host), so they ride the same apply==rebuild engine as P1.5a–d; only the `EditClass` differs.
- Module NodeIds are **path-derived** (`extract.rs::emit_crate` builds module ids from `module_path_segments` via `NodeId::from_components(["module", crate, seg…])`), so any rename/move/split changes every descendant module's id → ModuleTree class re-emits the whole editing crate's `contains`/module nodes (D3: `children_by_parent` = P, nodes/bindings = P) plus reverse-dep import bindings whose `use` paths changed.
- Split/merge validation is pure *is/ought*: the agent supplies the `Partition`; the engine only checks total coverage (every declared child item assigned exactly once) and name collisions — it never proposes a partition.
- Crate ops do `Cargo.toml` surgery with `toml_edit` (preserve formatting): `lift_to_crate` appends to root `[workspace] members` and adds a `path` dep on the consumer; identity flows through `loader::load_crate_target_kinds` (`crate_target_kinds_by_name`) and `emit_crate`'s `crate_id = NodeId::from_components(["crate", crate_name])`, so a new crate name = a new crate node. CargoManifest is the D2 cold-rebuild class (feature/member unification can ripple workspace-wide) → after a crate op the host re-`load()`s instead of incremental patch.
- Cycle safety for `lift_to_crate`/`move_module`: before writing, query `OpenedSnapshot::crate_edges()` (query/crates.rs) and reject any plan that introduces a back-edge (Principle 3 hard refusal, surfaced as `Refusal`/`RefusalReason` to the agent).
- Every op is wrapped in a D4 `Checkpoint` (rmc-host `UndoLog` marker + jj op id + RA edit-seq); the integration test diffs incremental-apply graph against a cold rebuild of the same final tree to catch silent ModuleTree-patch corruption.

**Risks / unknowns:**
- File-based-module surgery is multi-file and ordering-sensitive (create child file before inserting its `mod` decl, remove decl before deleting file); a half-applied plan must roll back cleanly through the D4 undo log — transactionality across N file writes is the main correctness hazard.
- `move_module` / `lower_to_module` cross-crate path rewrites (`crate::x` ↔ `othercrate::x`) reuse RA-assisted rename, which is **preview-only today** (per implementation plan) — applying and handling RA refusals (keyword/foreign-item conflicts) is unproven for module-granularity moves. (verify)
- CargoManifest → cold rebuild makes crate ops the slowest CRUD verbs (full `loader::load`); acceptable per right-sized rigor (rare, episode-scale) but means `simulate` (P1.4) for crate ops cannot reuse the cheap incremental path.
- `toml_edit` is not yet a workspace dependency — must be added to root `Cargo.toml [workspace.dependencies]` (only `toml = "0.9"` present today). (verify)
- Re-export-façade strategy after `split_module` (keep parent re-exports vs force callers to new paths) interacts with dead-pub (P13) and surface-stability (P12) penalties — partition semantics need a decision so reward isn't gamed by churning module names.
