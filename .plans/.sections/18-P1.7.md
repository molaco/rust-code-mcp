### P1.7 — Commit + reward computation

**Goal:** Turn `commit()` into the reward producer — snapshot audit + graph metrics pre-op, run the two-tier correctness gate (RA diagnostics per step / cargo at `declare_done`), diff audit + graph-metric deltas into a fixed-scalarization `RewardVector`, and roll the working snapshot back via the D4 checkpoint on any hard-fail.

**Key logic / algorithm:**
- **Two-tier gate (M0 Spike 2 fallback, the likely branch).** Per-step CRUD `commit` runs `RaGate::diagnostics` over the dirty `FileId`s on the already-warm host — near-free, type-level only. The real correctness gate (`CargoGate::check` + scoped `cargo test`) runs only at `declare_done` (P1.8), via `nix develop ../nix-devshells#cuda-code --command cargo …` per AGENTS.md. `GateTier::{PerStep, EpisodeEnd}` selects which runs. Borrowck/trait errors only surface at episode end — accepted and shaped for, exactly as Spike 2 prescribes; if Spike 2 reports `t_warm_check < ~2s` on T2 the per-step tier swaps to cargo-check with no type changes.
- **Audit delta = before/after scalar rollup.** `capture_audit_snapshot` calls the existing audits unchanged: `recursion_check(snap, opts)` for cycle count, `snap.dead_pub_report()` summed for dead-pub, `unsafe_audit(...)` filtered against the boundary allowlist (P1.6), `fn_body_audit(loaded, snap, opts)` for body-violation count, and `snap.crate_dependency_metric()` → mean `instability`/`abstractness` (the `CrateMetric` fields). `diff_audit` subtracts. Per the is/ought discipline, reward is keyed on these *deltas improving*, never on "acted on a flagged item." Per-step scope is the dirty crate only (fast); `declare_done` is workspace-wide.
- **Graph metrics over petgraph (new dep).** `capture_graph_metrics` builds a `petgraph::Graph` from `crate_edge_graph()` (crate-level) and `module_edge_pairs()` (module-level) — combinatorial, CPU, O(V+E), cheap at workspace scale (design: petgraph for graph algos, Candle stays for embeddings). Compute modularity (resolution-corrected Newman), conductance (boundary/volume), clustering coefficient (transitivity), Brandes betweenness-max. These score state *after* the move and feed `graph_metrics_delta`; they are reward shaping, not an objective the agent reads (Goodhart hardening deferred until gaming is observed).
- **Hard-fail → D4 rollback.** On `GateFailure` (compile break, scoped-test failure, or an escalated hard `Refusal`), `commit` calls `host.restore(cp)` (P0.3's atomic four-layer reset to the pre-op `Checkpoint`) so no partial source/graph/host state survives, and returns `CommitOutcome::HardFailed { restored_to, failure }`. The per-op `Checkpoint` is taken by the CRUD verb (P1.5a) before apply; `commit` only needs the handle.
- **Delta computation is incremental-friendly.** The pre-op `AuditSnapshot`/`GraphMetrics` are captured once at episode start (or carried from the prior step's after-state), so `commit` re-derives only the after-state over the dirty scope — avoiding the full-recompute latency wall the design warns about for per-commit metric deltas.

**Depends on:** [P0.2 (WorkspaceHost, OpenedSnapshot access, dirty FileIds), P0.3 (Checkpoint, host.restore — D4 rollback), P1.6 (GuidelineGate, Refusal, boundary allowlist feeding GateFailure/PrincipleViolation), P0.4 (a benchmark pool whose crates compile at baseline so the cargo gate is meaningful)].

**LOC:** prod +997 / test +705.

**Risks / unknowns:**
- **Cargo gate latency (Issue #2, second latency monster).** `cargo check`/`test` per commit is seconds-to-minutes; the whole design here assumes Spike 2's two-tier fallback (RA per step, cargo at `declare_done`). If even episode-end cargo is too slow on T2, commit batching or mid-episode error tolerance must be revisited. The `nix develop` wrapper adds shell spin-up overhead on every invocation — measure whether to keep a long-lived devshell process.
- **Graph-metric delta must stay incremental.** A naive full `crate_edge_graph()` rebuild + Brandes betweenness per commit re-creates the latency wall; betweenness is O(V·E) and is the costliest term — may need to cap it to crate-scale only (small V) or recompute it lazily (per-episode, not per-step) while modularity/conductance run per commit.
- **Scalarization weights are guesswork (verify).** `RewardConfig` is fixed per phase but the relative weights (gate vs audit-delta vs metric-delta vs length) are unknown until P1.8 rollouts show what dominates the gradient; credit assignment (commit-time compile break vs the 5 prior moves) is P1.8's problem but the reward shape constrains it.
- **RA diagnostics coverage (M0 Spike 2 coverage probe).** RA catches type-mismatch/missing-field/unresolved but misses borrowck and may miss trait-bound errors — the per-step gate is a type-level filter only, not a soundness guarantee. The reward must not over-trust a "type-clean" per-step signal; the hard gate remains cargo at episode end.
- **`unsafe_audit` vs boundary allowlist coupling.** The allowlist (P1.6, human-curated TOML) must be subtracted from the unsafe count before it enters `AuditSnapshot`, else allowlisted FFI shows as a standing violation and the delta is noisy — needs the P1.6 allowlist loader wired into `capture_audit_snapshot`.
- **`tests_passed` as a float.** The design shows `tests_passed: 0.98`; parsing `cargo test` libtest output (`N passed; M failed`) is brittle across harnesses (nextest vs libtest, doctests) — `CargoGateError::ParseOutput` covers failure but the ratio semantics need pinning to one runner.
- **Partition definition for modularity/conductance (verify).** Whether the partition is crates, modules, or the P1.3 clusters changes the metric meaning; clusters aren't stable enough to gate reward early, so crate/module partition is used first — but that may under-credit intra-module refactors.
