**Key logic / algorithm:**
- Body-span resolution reuses the proven skeleton machinery: `Node.span: Option<(u32,u32)>` bounds the item; `skeleton::source::find_item_syntax(parsed, node)` returns the `SyntaxNode`; the new `fn_body_range` casts `ast::Fn` and takes `body()?.syntax().text_range()` — the exact range `collect_body_replacements` substitutes `{ /* ... */ }` into (source.rs:265-296). Splice = `String::replace_range` over that window, then `SourceFile::parse(.., Edition2024)` must report zero errors or we return `ModifyBodyError::Reparse` before any host mutation (mirrors the parse-error guard at source.rs:127-134).
- Edit is classified `EditClass::BodyOnly` (signature/visibility/items provably unchanged because only the block expr range was touched) and handed to `WorkspaceHost::apply` (P0.2). Per D2/D3 this is the cheapest affected set: re-extract the editing fn's crate via `emit_crate_scoped`, patch ONLY `usages_by_id` + `usages_by_consumer_function` (the touched fn) + `usages_by_target`/`usages_by_consumer` secondaries + the `meta_by_key` usage count; `nodes_by_id`, `bindings_*`, `children_by_parent`, `signatures_by_target` stay unchanged. No reverse-deps. The host returns a `GraphPatch` whose inverse seeds the D4 undo log.
- Gate hook (P1.6): after the patch lands on the working snapshot, run `graph::fn_body_audit::fn_body_audit(loaded, snap, FnBodyAuditOpts { crate_id_filter: Some(crate_of(target)), patterns: ALL_PATTERNS, skip_test_fns: true })` scoped to the dirty fn — `unwrap`/`expect`/`panic_macros`/`unbounded_loop`/etc. on the new body become hard `Refusal`s; complexity/LOC/nesting thresholds layered by P1.6. A refusal rolls the patch back via the D4 undo marker and returns `CrudOutcome::Refused` (no source left mutated).
- Reward hook (P1.7): on a clean gate, `commit` diffs pre/post audit findings + graph-metric deltas into a `RewardVector`; `modify_body` returns `CrudOutcome::Applied { patch, reward }`. This is deliberately the lowest-cascade verb so the first loop exercises apply+gate+reward without any propagation correctness risk.

**Risks / unknowns:**
- `find_item_syntax` resolves by item_kind + display_name + recorded span; a stale span after a *prior* same-episode edit to the same file shifts offsets — must re-read source post-apply and rely on the warm-host re-extract refreshing `Node.span`, else the second body edit targets a wrong range. (verify span-refresh on the BodyOnly patch — D3 marks `nodes_by_id` "unchanged" for BodyOnly, so the stored span does NOT move; offset rebasing must come from re-parsing live source, not the graph.)
- Trait-default and `impl` method bodies share the `ast::Fn` path but the enclosing-item span vs body span interplay (nested fns, closures) needs the same non-overlap care `non_overlapping_replacements` already takes — a closure body inside the target fn must not be mistaken for the fn body.
- A body edit can change *outgoing* call edges (added/removed callee) → `usages_by_consumer_function` for this fn must be fully recomputed, not diffed naively; getting the DUP_SORT delete+reinsert right is the only correctness-sensitive patch here (Issue #5).
- Whether BodyOnly truly never ripples: a body edit that changes an inferred return via `-> impl Trait` desugaring could touch signatures; P1.5a guards by rejecting edits where the post-parse fn signature text differs from the pre-edit signature text (cheap string check) and escalating to the Sig/vis class if so. (verify against RA `impl Trait` cases.)
