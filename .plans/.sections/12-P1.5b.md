**Key logic / algorithm:**
- **move_item.** (1) classify the edit as `EditClass::ItemAddRemove` (D2) — a `pub` decl leaves one module and enters another, so the affected set is *editing crate (nodes/bindings/contains) + reverse-dep usages/bindings whose surface changed*; gather reverse-deps from the reversed `crate_edges()` consumer→producer graph plus `who_uses_targets([target])`. (2) `cut_decl_text` reads `Node.file`+`Node.span` from the working snapshot; `insert_decl_and_use` writes the decl into the dest module's file and inserts a `use <new_path>;` in the old module so unqualified local references still resolve. (3) Call `WorkspaceHost::apply_edits` (P0.2) with the dirty files; then `move_decl_references` runs RA's `analysis.rename`/`SourceChange` path (rename.rs) — repurposed not to rename but to recompute and rewrite fully-qualified `use`/path references across reverse-dep files, applied via the new `apply_source_change`. (4) Re-extract the affected set via `emit_crate_scoped` and `compute_patch`/`apply_patch` (P0.2 / rmc-graph). (5) Gate: `assert_no_new_cycle` compares `crate_edges_scc()` before vs after — if a multi-crate SCC appears that was not present pre-move, refuse with a `RefusalReason::Cycle{principle:"P3"}` and roll back the working-snapshot patch via the D4 undo log.
- **delete.** (1) `who_uses_targets([target])` (plus `who_imports`) collects all referencing consumers. (2) If non-empty and `cascade == false` ⇒ `DeleteError::StillReferenced` → structured `Refusal` (suggested_actions: `inline`, `move_target`, `delete_dependents_first`). (3) If `cascade`, transitively collect dependents (bounded reverse-dep walk over `who_uses_targets`), delete each decl + strip now-dangling `use` lines (reuse `imports_of` to find them), classify as `ItemAddRemove`, build the union affected set, patch LMDB.
- **Cycle check substrate.** `crate_edges()` already returns every cross-crate consumer→producer edge keyed by crate name; SCC over it is the canonical P3 detector. petgraph `tarjan_scc` is the new dependency (per design "petgraph for graph algos"); a CPU fallback exists in `graph::recursion_check::find_cycles_from`/`canonicalize_cycle` if petgraph is deferred (verify which lands).
- **Propagation reuse.** RA rename today is *preview-only* (`source_change_to_preview` in rename.rs builds a `RenamePreview` but never writes); this step adds the apply half (`apply_source_change`) and is the first consumer of it — `modify_signature` (P1.5c) reuses the same applied-`SourceChange` plumbing.
- **Differential safety.** Per cross-cutting P1.5 issue #3, the integration test runs each op then re-extracts cold (`graph::extract::extract`) and asserts the patched working snapshot's node/binding/usage sets equal the cold rebuild's.

**Risks / unknowns:**
- RA `rename` was built to rename a *symbol*, not to recompute import paths after a *move*; using `SourceChange` to fix `use` paths may require synthesizing a small AST edit instead, or a path-rewrite pass independent of `analysis.rename` (verify the RA API supports "item moved to new module, fix imports" — it may not, in which case `move_decl_references` becomes a hand-rolled `use`-path rewriter over `imports_of`/`usages_of`). This is the biggest unknown in the step.
- `apply_source_change` is genuinely new behavior on a preview-only surface; RA can refuse (keywords, foreign items, conflicting edits) — must surface those as `MoveError::RaRefused` not a panic.
- Cascade-delete blast radius: a transitive reverse-dep walk can touch many crates; bound it and prefer refuse. Default is refuse (open decision #2 in v3 leans `(a) refuse`).
- SCC over `crate_edges()` only catches *crate-level* cycles; an intra-crate module cycle from a move is not a P3 crate cycle but may still be undesirable — out of scope here (verify P3 scope is crate-level only).
- `Node.span` is `Option<(u32, u32)>` (byte offsets, verify) — decl-text extraction must handle attributes/doc-comments preceding the item (the `attributes` Vec is separate from the span) so `cut_decl_text` doesn't orphan `#[...]` lines.
- Reverse-dep affected-set correctness is load-bearing for training signal; covered by the differential test but the union-of-affected-sets logic for cascade is fiddly.
