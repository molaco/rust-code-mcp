**Key logic / algorithm:**
- The corpus exists to feed the *real* pipeline: every selected workspace must satisfy `graph::loader::load(dir)` → `graph::extract::extract`/`emit_crate` (loader.rs uses `ra_ap_load-cargo::load_workspace_at` + `CargoConfig`/`LoadCargoConfig`, so a workspace that can't resolve cargo deps can't be loaded, embedded, or gated). Therefore the gate is `cargo check` clean, not just "downloads."
- `verify::cargo_check` shells out through `DevshellRunner` (`nix develop ../nix-devshells#cuda-code --command cargo check --workspace --message-format=short`) per AGENTS.md; non-zero exit or any `error[` line → `passed=false` → entry retiered to `Excluded` and dropped from selectable pools (mitigates Issues-register #10: a non-compiling baseline makes the P1.7 cargo-gate reward meaningless).
- Pinning is content-stable: `Source::Git { url, rev }` checks out an exact commit; `CratesIo { version }` resolves an exact `=version`; `fetch_entry` is idempotent (no-op if the on-disk checkout already matches the rev) so `ensure()` is re-runnable and reproducible across machines.
- T2 selection (`sizing::pick_t2`) ranks `cargo check`-passing workspaces by `walkdir`-counted SLOC and picks the one closest to ~100k LOC; this is exactly the T2 the M0 spikes judge their 500ms (Spike 1) and ~2s (Spike 2) gates against, and where `cargo check` baseline timing is established for P1.7.
- The corpus root lives outside the rmc workspace `target/` and is git/jj-ignored; only `corpus.toml` (the pinned manifest) is checked in, so CI/other machines reproduce the identical corpus by re-running `rmc-bench fetch`.

**Risks / unknowns:**
- Many crates won't `cargo check` clean under `cuda-code` (nightly-only features, system/native deps, `build.rs` toolchains absent in the devshell). Must over-fetch (e.g. 120–150 candidates) to net 50–100 passing; filter hard.
- Single ~100k-LOC virtual workspace vs an aggregate-to-100k bundle: a real large workspace is the cleaner T2, but availability under the devshell is uncertain — `pick_t2` may need to allow a synthetic multi-crate aggregate (verify whether `loader::load` tolerates that vs. a true single-workspace manifest).
- `cargo check --workspace` over 50–100 corpora is minutes-to-hours wall-clock on first run; needs a persistent per-corpus `target/` cache and parallel `verify_all` (rayon) — but parallel cargo invocations contend for the shared registry/lockfile.
- Shelling `nix develop … --command cargo …` per crate has nontrivial devshell-eval startup cost; may want one long-lived `nix develop --command bash -c "…"` session (verify feasibility vs. capturing per-crate diagnostics).
- xtask-vs-crate choice: spec'd here as the `rmc-bench` crate (workspace member with a bin), not a root `xtask`, to keep the canonical crate set; if the team prefers `xtask` the file table maps 1:1 onto an `xtask/` bin.
- `rev` drift / upstream force-push or yanked versions break reproducibility; mitigate by recording a content hash alongside `rev` in `CrateEntry` (the `sha2` dep is for this).
