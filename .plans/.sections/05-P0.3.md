### P0.3 — Checkpoint/restore (D4) + jj rollback
**Goal:** Make a `Checkpoint` a first-class four-layer contract (jj op id + graph undo-log marker + RA host edit-seq + content-hash caches) and `restore()` reset all four atomically, sub-second, so per-op CRUD wrapping and `rollback()` never expose partial state.

**Key logic / algorithm:**
- **Undo journal feeds off the apply path (D4 "per-op undo log, not full LMDB copies").** Before each LMDB mutation in P0.2's `apply_patch`/`compute_patch` (which `put`s into the same tables `write_model` does — `nodes_by_id`, `bindings_by_*`, `children_by_parent`, `usages_by_*`, `signatures_by_target`, `static_metadata_by_target`, `meta_by_key`), call `capture_prior` and push an `UndoEntry`. `checkpoint()` just records `undo.mark()` — O(1), no copy. Embeddings/descriptions caches are content-hash keyed (D3 `C`), so they are *not* journaled — they self-heal on next read.
- **`restore_atomic` resets the four layers in dependency order, all-or-nothing:** (1) `vcs::op_restore(cp.source)` rewinds source files to the checkpoint jj operation; (2) open one working-snapshot `wtxn`, `undo.replay_to(cp.graph)` inverts every LMDB patch back to the marker, then `wtxn.commit()` (single transaction = graph atomicity for free, mirroring `write_model`'s one-txn commit); (3) re-sync the warm RA host by replaying inverse `set_file_text` to `cp.ra_edit_seq` over the same dirty files (reuses P0.2's `set_file_text` ingestion); (4) caches need no action. Then `undo.truncate_to(cp.graph)`.
- **Atomicity escape hatch (D4 "a failed restore re-opens the working snapshot from base").** If undo replay or host re-sync fails, abort the partial reset and re-init the working snapshot from the base published snapshot via the P0.2 `mdb_copy`-from-base path + re-open RA host from base, then surface `RestoreError::FellBackToBase`. Slow path, explicitly avoided in the happy case.
- **jj wrapper stays thin and subprocess-based** (jj 0.40.0 confirmed present; `jj op log`/`jj op restore` are the primitives). It only knows source; the contract's value is that P0.3 binds it to graph+host+caches so `rollback()` is never "just a jj wrapper" (the design's explicit warning).
- **Working-snapshot identity is respected (D1).** Restore decrements the host's `edit_seq` back to `cp.ra_edit_seq`; identity `(session_id, base_graph_id, edit_seq)` is content-decoupled, so reversing patches in place is legal — no new content-addressed `graph_id` is minted (unlike published snapshots in `storage::graph_id_for`).

**Risks / unknowns:**
- jj op-id template: `jj op log -T 'id.short()'` flag/format must be verified against jj 0.40.0 (verify) — fallback is parsing the leading hex token of default `jj op log --no-graph` output (the format observed in this repo).
- RA host inverse-`set_file_text` replay may not reproduce the exact pre-edit salsa state if intermediate edits touched the same file non-monotonically; the D4 fallback (re-open from base) covers correctness but is slow — needs the P0.2 fan-out measurement to know how often it triggers.
- DUP_SORT inverse precision: deleting the *one* `(key,val)` pair we added (not the whole duplicate set) requires heed `delete_one`/cursor semantics; a wrong delete silently corrupts secondaries → corrupts reward (mirrors implementation issue #5). Differential test (restore == cold-rebuild-of-pre-state) on every op is the guard.
- `jj op restore` rewinds the *whole working copy*; if the episode is the only writer this is fine, but concurrent rollout workspaces must each be a distinct jj workspace/op-head (verify P0.4 pool gives one jj repo per workspace clone).
- Counter inverses (`meta_by_key` usage/node counts, D3 `P`) must be journaled too, else manifest/`meta` counts drift after rollback — easy to miss; covered by `MetaCounter` variant.
- `Instant` in `Checkpoint` is non-serializable; if checkpoints must persist across a process restart for crash-recovery, swap to a logical seq (out of scope here — checkpoints are in-session/ephemeral like the working snapshot).
