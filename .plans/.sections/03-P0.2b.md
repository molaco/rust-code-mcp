### P0.2b — Scoped re-extract + affected-set (D2)
**Goal:** Classify a CRUD edit into an `EditClass`, compute the affected crate set via reversed `crate_edges`, and re-run only the affected crates' extraction passes against the warm `RootDatabase` to produce a scoped `ExtractionModel` ready for diff-patch.

**Key logic / algorithm:**
- **Scoped re-extract.** Lift the per-crate loop body in `extract()` (extract.rs:73-88) into `emit_crate_scoped`: call the existing private `emit_crate` for each crate in the affected subset, then call the existing passes (`extract_bindings`, `extract_impl_items`, `extract_attributes`, `extract_signatures`, `extract_statics`, `extract_usages`) with `local_crates` narrowed to the affected slice. Crucially the warm `db` still holds the *full* crate graph, so cross-crate resolution inside `extract_bindings`/`extract_usages` (which call `Definition::usages(sema).all()`) stays correct — only the *iteration domain* (which crates' modules/items we walk) shrinks. The `def_to_node` map is rebuilt for the affected crates only.
- **EditClass → base crates.** `classify()` follows the D2 table: BodyOnly→editing crate only, no reverse-deps; SignatureOrVis / ItemAddRemove / ModuleTree→editing crate + direct reverse-deps; Macro→editing crate + transitive reverse-dep closure (generated code opaque); CargoManifest→`full_rebuild=true` (cold rebuild, no scoping).
- **Reverse-deps from crate_edges.** `crate_edges()` (query/crates.rs:123) already yields `CrateEdge{consumer_crate, producer_crate, ...}`. `reverse_crate_edges()` inverts to producer→[consumers]; `affected_set()` looks up the edited crate's qualified name, takes its consumers (direct) or BFS closure (Macro), maps names back to crate `NodeId` via the `NodeKind::Crate` scan already present in `crate_edges`.
- **Partition by crate for the diff.** Every `Node`/`Usage` carries `crate_id` (model.rs:95) and `Binding.from_module`/`target` resolve to crates via the `node_to_crate` map (query/crates.rs:139) — so the produced scoped `ExtractionModel` can be partitioned per affected crate, which is exactly the input the LMDB diff-patch step (separate step, D3 matrix) consumes. This step stops at producing the scoped model + the affected-set; it does not write LMDB.
- **No-textual-diff classifier.** Per D2, `EditDescriptor.class` is supplied by the CRUD op (it knows it changed only a body vs a signature). `classify()` never re-derives the class from source, avoiding a fragile diff parser.

**Depends on:** [P0.2a] (warm `WorkspaceHost` owning the live `RootDatabase`/`Vfs` and the `set_file_text` edit path; this step consumes that warm `db` to re-extract).

**LOC:** prod +538 / test +395.

**Risks / unknowns:**
- **Pass-scoping correctness.** `extract_bindings` seeds `def_to_node` from *all* `module_node_for` (bindings.rs:42-44); narrowing the crate set means a consumer crate re-extract must still see producer crates' module/item NodeIds to resolve imports. Likely requires keeping a *read-only* full `def_to_node`/`module_node_for` from the base model and only re-walking the affected crates' own scopes — needs validation against the cold-build golden (P0.1).
- **Reverse-dep granularity is crate-level, not item-level.** D2 says Sig/Vis affects "reverse-deps *that reference the item*", but `crate_edges` is crate-granular. First cut re-extracts whole reverse-dep crates (over-approximation, still correct); item-level pruning is a later optimization. (verify whether per-symbol `EdgeSymbol.target_qualified` can cheaply prune.)
- **Macro transitive closure size.** On a hub crate (`rmc_graph`) the reverse-dep cone may be most of the workspace — Macro/CargoManifest edits collapse toward cold rebuild; acceptable per right-sized rigor but must be measured (M0 Spike 1 hub case).
- **`emit_crate` arg-bundling churn.** Collapsing 11 args into `EmitCrateCtx` touches `extract()` and the existing extract.rs tests; mechanical but must preserve the exact insertion order (`crate_node_for`/`crate_name_for` populated in `emit_crate` and consumed by later passes).
- **ModuleTree FileId churn.** Adding/removing a `mod` (new `FileId` + source-root change) is the hardest warm-host edit (m0-spikes notes the API is harder); the affected-set logic is unchanged but the upstream P0.2a edit ingestion must supply a correct `touched_files`/new-file signal.
