**Key logic / algorithm:**
- The warm-mutation primitive is confirmed against pinned `ra_ap_*` 0.0.330: `RootDatabase::apply_change(ChangeWithProcMacros)` (ide_db `apply_change.rs`), `ChangeWithProcMacros` re-exported from `ra_ap_hir`, wrapping `FileChange::change_file(FileId, Option<String>)` (base_db `change.rs`). `RootDatabase: Clone` (salsa snapshot) — recorded for the D4 checkpoint step, not used here.
- Lifecycle change: today `snapshot::build_and_persist` calls `loader::load` → `extract::extract(&loaded)` and drops `loaded` (its `RootDatabase`) immediately. `WorkspaceHost::open` keeps that `RootDatabase` + `Vfs` alive for the session; `extract_from` lets P0.2b re-extract dirty crates off the *same* warm db via the existing per-crate `emit_crate`.
- Dirty-file ingestion: caller passes absolute paths; `file_id` resolves them through `Vfs::file_id`; `apply_edits` batches `change_file` calls so a multi-file CRUD op is one salsa invalidation. `edit_seq` increments per applied batch and is the marker D4 restores to via inverse `set_file_text` replay.
- Proc-macro survival (the explicit FOCUS unknown): the current `LoadCargoConfig` sets `with_proc_macro_server: ProcMacroServerChoice::None` and discards the `_proc_macro` handle, so **no proc-macro server exists warm today**. `load_with_proc_macros` plumbs `ProcMacroServerChoice::Sysroot` and *retains* the handle inside `WorkspaceHost`; a spike test (Spike 1 Step 0) asserts a `#[derive]`/`macro_rules!`-heavy item still resolves after a warm `apply_change`, otherwise the finding is "macro edits force re-`load()`" (D2 already routes Macro class to full rebuild, so this is a constraint check, not a blocker).
- Channel-actor (mpsc + oneshot) wrapper is deferred: `WorkspaceHost` is `&mut`-driven and single-threaded here; the actor lands with the episode loop (P1.8) if concurrent rollouts need it. Noted, not built in P0.2a.

**Risks / unknowns:**
- `ChangeWithProcMacros` constructor surface: it re-exports from `ra_ap_hir` but the struct/its `set_proc_macros` field is defined deeper (hir_expand/base_db); exact construction path needs a compile-check (verify).
- Proc-macro server may tear down or not survive `apply_change` warm (FOCUS item) — measured by the Spike 1 Step 0 gate; if it dies, macro-touching edits fall back to cold reload.
- Memory: a warm `RootDatabase` holds the whole crate graph + types; one per concurrent rollout workspace multiplies footprint (issue #4 in the register).
- `local_crates: Vec<Crate>` caches `CrateId`-like handles that may be invalidated by a crate-graph-changing edit (module-tree/Cargo); `reload_crates` covers the common case but Cargo edits are cold-rebuild (D2) anyway.
- VFS/db lockstep: editing `db` via `change_file` without mirroring `vfs.set_file_contents` would desync path→FileId lookups for later edits; `set_file_text` updates both — must stay paired (verify no double-apply).
