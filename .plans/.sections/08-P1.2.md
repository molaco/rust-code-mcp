**Key logic / algorithm:**
- Cache substrate reuses the embedding precedent exactly: new `descriptions_by_target` sub-DB keyed by `NodeId::as_bytes()` with value `DescriptionRecord`; cache key is `(content_hash, model_id)` where `content_hash = SHA-256(trimmed item source slice)[..16]` — copy `prepare_embeddings_for`/`write_embedding_cache` (`embedding_cache.rs`) into `description_cache.rs`, swapping the embedder call for the LLM call. Adding the DB requires `SCHEMA_VERSION 12→13` (auto-rebuilds old snapshots via `graph_id_for`) and raising `DEFAULT_MAX_DBS` (currently 13 DBs created against a cap of 16).
- Prompt assembly calls into `skeleton::source::render_source_item_text` (body-stripped declaration, design principle 4 "bodies hidden") plus `snap.function_signature` and 1-hop neighbors from `usages_by_consumer`/`usages_by_target` so the description captures intent ("normalize a path") not just the symbol name (`dir_hash`).
- Merkle-keyed regen: `FileSystemMerkle::detect_changes` yields the changed file set; map files→items and regen only those whose recomputed `content_hash` no longer matches the stored row — same self-invalidating rule as embeddings (D3: `descriptions = C`). Per-item content_hash (not just file membership) is the true invalidation key, so an unedited item in an edited file is reused.
- Retrieval: embed the query string via `EmbeddingGenerator::embed_queries` and cosine (reuse `graph::math::cosine`) against the optionally co-stored `DescriptionRecord.embed_vector`; this makes descriptions the 4th retrieval axis ("what is *described as* doing X"). BM25 over the raw description text via Tantivy is the feature-gated fallback (Tantivy already a workspace dep).
- Generator is a standalone reqwest client modeled on `embeddings/openrouter/client.rs` (chat/completions instead of /embeddings), keeping the description sub-model off the main-agent compute path (open decision 3, lean separate).

**Depends on:** [P0.1 (determinism/content-hash stability), P0.2 (dirty-set for refresh_dirty), P1.1 (view consumes descriptions); reuses rmc-engine EmbeddingGenerator and rmc-graph skeleton/embedding_cache patterns].
**LOC:** prod +1203 / test +650.

**Risks / unknowns:**
- Generation cost/throughput at workspace scale (thousands of items × LLM round-trips); batching + concurrency cap mitigate but episode-0 cold gen may be minutes — acceptable since descriptions are once-per-workspace + dirty-incremental, not per-step.
- Staleness within an episode (issue #8): CRUD edits items but regen lags; mitigated only if `refresh_dirty` rides the P0.2 dirty set — otherwise the agent navigates on stale labels.
- Model choice still an open decision (separate small model vs main agent); `model_id` is baked into the cache key so a model swap auto-invalidates (good), but a wrong/cheap model yields low-utility descriptions that distort the view.
- `descriptions = C` in D3 is per-content-hash; must confirm content_hash here uses the **same** trimmed-slice rule as `embedding_cache.rs` so the two caches invalidate in lockstep (verify against actual span/trim handling).
- Co-storing `embed_vector` inside `DescriptionRecord` couples description regen and retrieval-embedding regen; alternative is reusing `embeddings_by_target` over description text in a separate table — decide before schema is frozen (verify).
- Exact rmc-server tool-registration path and signature not yet read (verify path); whether retrieval is LanceDB vs in-memory cosine vs BM25 is left configurable but the default needs measuring.
