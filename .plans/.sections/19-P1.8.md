### P1.8 — Episode runner + trajectory logging
**Goal:** Close the loop — define the top-level `Action` enum over the five verb categories, an `EpisodeRunner` that drives view→action→reward under a `StepBudget`, a `Trajectory`/`TrajectoryStep` log (the future SFT data format), and an API harness that runs a frontier model end-to-end on the rmc workspace itself.

**Key logic / algorithm:**
- **Action is composition, not the enum.** `Action` is exactly the five canonical sub-enums (`nav::Navigate`, `AnalyzeOp`, `QueryOp`, `crud::Crud`, `MetaOp`) under one tag — the "alphabet," not a "phrasebook." The runner never decomposes a labeled intent; the policy emits primitives and sequences them (design "Composition vs the Enum"). The whole enum is `Serialize`/`Deserialize` so it doubles as the policy's action schema and the trajectory's logged action with no separate DTO.
- **The loop is the design's Episode Shape, verbatim.** `run` mirrors the v3 pseudocode: `obs = build_observation` (P1.1 `ContextView` + cached P1.3 `VisionMap`/`Outlier` + P1.2 `Description`); `action = policy.choose(obs, available)`; navigate/analyze/query → `StepResult::Observed` + budget debit (the observation cost = "budget pressure"); Crud → `CrudEngine` returns `CrudOutcome::{Applied,Refused}` (P1.5a `modify_body` is the only wired verb at M3; b–e widen it); `Meta::Commit` → P1.7 full `RewardVector`; `Meta::DeclareDone`/budget-exhaustion → break with `EpisodeOutcome`. Most trajectories naturally become "attempt body → blocked by gate → must restructure," the trained skill.
- **Per-episode-once vs per-step (right-sized rigor / latency).** `EpisodeRunner::run` computes the Analyze layer ONCE at episode start (`ClusterEngine` + `outliers` + `affinity` + `VisionMap`, P1.3) and the description index once, caching both in `EpisodeState`; the per-step cost is only the policy forward pass + the cheap incremental view assembly + (on CRUD) the P0.2 sub-second re-extract. This honors the "run expensive global stuff less often than cheap local stuff" rule without any enforcement machinery.
- **Every step is checkpointed; the trajectory is the audit log.** Before each CRUD/commit the runner takes a D4 `Checkpoint` (P0.3) and records its `CheckpointId` into the `TrajectoryStep`, so `Meta::Rollback` is `WorkspaceHost::rollback` to a prior step and "how did this reach this state?" is answerable from the trajectory alone. A hard-fail commit auto-rolls-back (P1.7 contract). Refusals are first-class `TrajectoryStep`s (`StepRecord::Refused`) — every refusal is training data per the design.
- **SFT data format is locked by `trajectory_format.rs`.** `Trajectory` serializes to JSONL: one `TrajectoryStep` per line (streamed via `TrajectoryWriter::append` so partial runs survive a crash) + a trailing `EpisodeOutcome`. Each step carries the full `ContextView` (the prompt the policy saw), the `available: Vec<ActionKind>` (the legal-move mask), the chosen `Action`, and the `StepRecord` (reward / refusal) — exactly the (observation, action, reward) tuple SFT/RL consume. The golden test pins the schema so downstream training code does not break on a field rename.
- **Frontier-model harness.** `ApiPolicy` renders the structured-text view (P1.1 `render_context_view`) + the legal action list into a single chat turn, advertises the `Action` JSON/tool schema, and parses the reply back into an `Action`; `EpisodeRunner` is policy-agnostic (`dyn Policy`), so the same loop runs the `ScriptedPolicy` in CI (deterministic, no network — the M3 `episode_modify_body_e2e` test) and a real model via the `run_episode` MCP tool on the rmc repo's own `project_paths` dedupe (the M3 exit criterion).

**Depends on:** [P1.1 (Action/Navigate, Location, ContextView, render_context_view), P0.2 (WorkspaceHost, working snapshot, scoped re-extract), P0.3 (Checkpoint/rollback), P1.2 (DescriptionIndex), P1.3 (VisionMap/clusters/outliers cache), P1.4 (Simulator for Query::Simulate), P1.5a–e (CrudEngine + Crud verbs), P1.6 (GuidelineGate/Refusal/RefusalReason), P1.7 (RewardVector/RewardConfig/commit), P0.4 (workspace pool as the episode source)].

**Risks / unknowns:**
- **Credit assignment (Issue, design + register).** A commit-time compile/test break is attributable to the commit step but caused by ≤N prior CRUD moves; the trajectory logs per-step rewards + the episode summary, but the *scalarization across steps* (how the gate failure back-propagates) is a training-side decision deliberately left to Phase 3 — P1.8 only guarantees the data is captured, not the credit rule.
- **`ContextView` is heavy to log every step.** Embedding the full view per `TrajectoryStep` makes JSONL large; may need a `view_digest` (hash + delta-from-previous) instead of the full view for long episodes — decide before freezing the schema (the golden test would have to bless either form). (verify acceptable size on a 50-step episode.)
- **Expressiveness / abstraction leak (register Issue).** The five-verb set may not cover every refactor the rmc `project_paths` task needs (e.g. a cross-crate helper extraction that doesn't fit `extract_function`/`lift_to_crate`); the M3 run is also the test of whether the action space is complete — a stuck frontier model is a signal the verb set needs widening, not a runner bug.
- **`ApiPolicy` action parsing robustness.** Frontier models emit malformed/partial tool calls; `Unparseable` + single retry is the floor, but a model that never produces a valid `Action` stalls the episode — needs a max-reparse-attempt cap folded into `StepBudget` (else an episode burns the whole budget on parse failures). (verify the chat/tool-call schema matches the chosen provider; reuses the `embeddings/openrouter/client.rs` reqwest precedent, but that is /embeddings not /chat.)
- **`run_episode` MCP tool is long-running** (seconds-to-minutes per episode under the P1.7 cargo gate) — the existing `#[tool]` methods are request/response; an episode may exceed MCP client timeouts. May need to return a job handle + poll (like the background sync task in `ServerRuntime`) rather than block, or run episodes only via a dedicated `rmc-env` binary/`rmc-bench` driver and keep the MCP tool for short scripted demos. (verify MCP timeout behavior; the table assumes the simple blocking form.)
- **Concurrent rollouts multiply the warm `WorkspaceHost`** (memory, register Issue #4): one `EpisodeState` per concurrent episode holds a full RA `RootDatabase`. P1.8 runs single-episode; the mpsc+oneshot channel-actor wrapper for the host (deferred in P0.2a) lands here only if parallel rollouts are needed, and is noted as the scaling path, not built now.
- **`EpisodeState` ownership of `WorkspaceHost` vs `&mut` borrows by `CrudEngine`/`Simulator`.** `CrudEngine` (P1.5a) holds `&mut WorkspaceHost` and `Simulator` (P1.4) holds `&WorkspaceHost`; threading both through one `EpisodeState` per step without borrow conflicts likely forces the host behind the channel-actor or an `&mut`-reborrow per dispatch arm — a borrow-checker ergonomics risk that may reshape the `dispatch` signature. (verify the engines can be reconstructed cheaply per step or must be long-lived.)
