# THEORY_2.md

Operational theory for AI agents doing structural and type-level refactors in
this Rust workspace. This document is a companion to `THEORY.md`. It keeps the
same practical goal, but makes the category-theory and HoTT connection explicit
enough to guide decisions without turning the workflow into abstract math.

## §1. Premise

The workflow has two layers:

1. **Structural layer.** Files, modules, crates, and workspaces form dependency
   graphs. Category theory is the useful lens here: units are objects,
   references are morphisms, chains of references compose, and refactors change
   how the graph is factored into containers.

2. **Type/function layer.** Structs, enums, traits, functions, and methods form
   contracts. HoTT is the useful lens here: types describe shapes of values,
   functions are transformations between shapes, and refactors are acceptable
   when the old and new shapes remain equivalent for the callers that matter.

The two layers meet at boundaries. A module or crate boundary is only useful if
the public surface is stable, named, and small enough for callers to depend on.
A type boundary is only useful if it states exactly the contract that callers
use, no less and no more.

This document is for agents. Use math names only when they create an executable
check. If a principle does not tell the agent what to inspect, move, split,
merge, lift, or lower, it is not useful here.

## §2. The Two Calculi

| Layer | Question | Math lens | Code units | Main tools |
|---|---|---|---|---|
| Structural | Where should code live? | Category theory | files, modules, crates, workspace | `get_dependencies`, `get_imports`, `module_tree`, `crate_edges`, `who_imports`, `who_uses_summary` |
| Type/function | What shape should code have? | HoTT / type theory | structs, enums, traits, fns, methods | `function_signature`, `who_uses`, `calls_from`, `who_calls`, `semantic_overlaps`, `fn_body_audit` |

Structural work comes first when the project layout is chaotic. Type/function
work comes after the boundaries are good enough that agents can reason locally.

The order matters:

1. Discover behavior quickly.
2. Refactor the dependency graph into understandable containers.
3. Refactor local code inside each container until the guidelines are enforced.

Skipping step 2 makes step 3 brittle. The agent is forced to reason about local
functions while the surrounding boundary graph is still unstable.

## §3. Category Theory as Boundary Calculus

For each level, treat the codebase as a finite directed multigraph and then as
the free category generated by that graph.

| Category | Objects | Generating morphisms |
|---|---|---|
| `C_file` | source files | imports, type references, fn calls |
| `C_module` | Rust modules | `use`, submodule relation, visible item references |
| `C_crate` | crates | cross-crate item references and Cargo dependencies |
| `C_workspace` | workspace members | member relation, crate dependency relation |

The identity morphism is the unit itself. Composition is a dependency path. If
`A` uses `B`, and `B` uses `C`, then `A` reaches `C` through a composed path.
Long composed paths are not automatically bad, but they reveal how change can
travel.

### Boundaries

A boundary is a partition of objects into containers. A good boundary has:

- high internal edge density,
- low external edge density,
- stable external morphisms,
- a public surface that can be named in one sentence,
- no dependency cycle at the level where a cycle is forbidden.

This is the categorical meaning of "directory layout." A directory is not good
because it has a pleasing name. It is good when it is a container whose internal
morphisms dominate its external morphisms.

### Quotients and Containers

When files are grouped into a module, the module is a quotient of the file graph:
many file objects are treated as one higher-level object. When modules are
lifted into a crate, the crate is a quotient of the module graph.

The quotient is useful only if the collapsed internal edges stay internal and
the remaining external edges form a reasonable public surface. If most edges
still cross the new boundary, the quotient is fictional.

### Functors Between Levels

Moving from files to modules or modules to crates is a functor-like operation:
objects map upward to containers, and morphisms map to either internal edges or
public boundary edges.

Valid lifting preserves the important graph facts:

- dependencies still point in the same direction,
- cycles are not introduced at the lifted level,
- the public surface is smaller than the internal graph,
- callers can still reach the needed symbols through stable names.

Lowering reverses the move: crate to module, module to file. It is justified
when the higher-level object has too little independent surface to deserve its
own boundary.

### Projection

Visibility is projection. `pub`, `pub(crate)`, `pub(super)`, and re-exports
decide which internal morphisms are projected onto the public interface.

The public interface is not all code inside the container. It is the projection
of the internal graph that outside callers are allowed to observe.

Tools:

- File imports: `get_dependencies(file_path=...)`.
- Module imports: `get_imports(directory=..., module=...)`.
- Module tree and file-backed modules: `module_tree(directory=..., krate=...)`.
- Public surface: `get_exports`, `get_declared_reexports`, `get_reexports`.
- Crate graph: `crate_edges`, plus Cargo metadata if Cargo dependency edges are
  the question.
- Boundary fan-in: `who_imports`, `who_uses_summary`.

Rust note: a module is a namespace, not always a file. Most modules are
file-backed, but inline modules mean one file can contain several module objects.
Use `get_dependencies` for exact file-level imports and `get_imports` for
module-level imports.

## §4. HoTT as Type and Function Calculus

HoTT is useful here as a discipline for deciding when two designs are equivalent
for callers, and when a new type shape is justified.

This document does not require formal proofs. The agent should use HoTT as a
practical vocabulary:

- A **type** is a shape of possible values.
- A **function** is a transformation from one shape to another.
- A **trait/signature** is the contract callers rely on.
- An **equivalence** means callers can move between old and new shapes without
  losing the behavior they depended on.
- A **path** is a migration route from old usage to new usage.

### Product Decomposition

Structs are product-like shapes: a value contains several fields at once.

Split a struct when independent groups of fields have independent lifetimes,
callers, invariants, or construction paths. Merge structs when the pieces are
always created, passed, changed, and validated together.

Operational checks:

- field groups co-change together,
- functions consume one field group without the other,
- constructors expose unrelated arguments,
- tests repeatedly build partial values.

### Sum Decomposition

Enums are sum-like shapes: a value is one variant among several.

Split an enum when callers only handle disjoint variant groups. Merge enums when
callers immediately convert between them or match them in lockstep.

Operational checks:

- `who_uses` on variants if available,
- `find_references` for variant names,
- duplicated match arms across modules,
- conversion functions that only translate between near-identical enums.

### Contract Extraction

A trait is honest only when it describes what callers actually use. The extracted
trait surface should come from the callsite-usage set, not from the full method
surface of the concrete type.

This is the HoTT reading of abstraction: the trait is the proposition callers
need proved. Extra methods are extra obligations. A trait that duplicates the
entire concrete type is usually not an abstraction.

Tools:

- `who_uses(target=...)` to find observed consumers.
- `calls_from(caller=...)` and `who_calls(target=...)` for method/function use.
- `function_signature(target=...)` for callable shape.
- `similar_to_item(target=...)` and `semantic_overlaps(...)` for duplicate
  shapes worth abstracting.

Known gap: current tooling does not provide a one-shot callsite-usage-set tool
for all methods used through a concrete type. The agent must combine references,
call graphs, and source inspection.

### Equivalence and Refactor Safety

A local refactor is safe when the new shape is equivalent for its callers:

- same reachable behavior,
- same required invariants or stricter invariants with migration,
- same or intentionally changed public signature,
- tests or type signatures witness the equivalence.

In mature code, equivalence must often be compatibility-preserving rather than
text-preserving. The old API can remain as an adapter while the new API becomes
the canonical shape.

## §5. The Three Primitive Operations

Every refactor in this workflow is built from three primitives:

1. Move.
2. Split / merge.
3. Lift / lower.

All specific operations are compositions of these primitives.

### O1. Move

**Definition.** Move an existing unit or subunit to a different container at the
same level.

Examples:

- move a file between module directories,
- move a function between files,
- move a type between modules,
- move a module directory between crates,
- move a re-export from one facade to another.

Categorical reading:

- The object remains conceptually the same.
- Its incident morphisms are reattached through a different container.
- The move is good when it turns crossing edges into internal edges.

HoTT reading:

- The term/type should remain equivalent for callers.
- Public paths may change, so adapters or re-exports may be needed.
- If callers observe the path, the path is part of the contract.

Preconditions:

- Most references point toward the destination container.
- The destination has a name that can honestly include the moved unit.
- The move does not introduce a cycle at crate level or an unacceptable module
  cycle.

Checks:

- `who_uses_summary(target=...)`
- `who_imports(target=...)`
- `get_dependencies(file_path=...)`
- `get_imports(directory=..., module=...)`
- `crate_edges(directory=...)`
- `re_export_chain(target=...)`

Failure modes:

- moving by name similarity instead of edge density,
- moving a hub without migration planning,
- hiding a cycle behind a facade,
- widening visibility to make the move compile.

### O2. Split / Merge

**Definition.** Change the number of units at the same level.

Examples:

- split one large file into several files,
- merge several one-file directories into one module,
- split a module into submodules,
- merge a thin module back into a file,
- split a crate,
- merge a small crate into a module,
- split a struct, enum, trait, function, or method.

Categorical reading:

- Split refines one object into several objects with explicit morphisms between
  them.
- Merge quotients several objects into one object, internalizing their edges.
- The correct split minimizes crossing edges and maximizes internal coherence.

HoTT reading:

- Product split: divide a struct into independent component types.
- Sum split: divide an enum into independent variant families.
- Function split: factor one transformation into named intermediate
  transformations.
- Merge: identify shapes that are always used as one shape.

Preconditions for split:

- The candidate contains multiple named surfaces.
- Internal groups are denser than edges between groups.
- Callers usually use one group at a time.
- The resulting public surfaces can be named.

Preconditions for merge:

- Separate units always change together or call each other directly.
- The boundary creates more imports than understanding.
- The smaller unit has no independent public surface.

Checks:

- `module_tree(directory=..., krate=...)`
- `get_exports(...)`
- `get_declared_reexports(...)`
- `semantic_overlaps(...)`
- `analyze_complexity(file_path=...)`
- `who_uses_summary(target=...)`
- `dead_pub_report(directory=...)`

Failure modes:

- creating one huge directory and many one-file directories,
- splitting by visual length rather than morphism density,
- merging away a bridge unit,
- splitting a type but leaving constructors and invariants tangled.

### O3. Lift / Lower

**Definition.** Change the abstraction level of a unit.

Examples:

- file to module,
- module to file,
- module to crate,
- crate to module,
- concrete type to trait,
- concrete type to wrapper/supertype,
- field group to child type,
- broad type to smaller types.

Categorical reading:

- Lift maps objects and morphisms into a higher category.
- Lower maps a weak higher-level object back into a lower category.
- A lift is justified when the higher-level object has a stable public surface.

HoTT reading:

- Concrete to trait: replace a specific inhabitant with the contract callers use.
- Type to wrapper/supertype: embed a value into a broader shape.
- Field group to child type: make an implicit invariant explicit.
- Lowering specializes a general shape when the generality is not used.

Preconditions for module to crate:

- The module has a coherent named surface.
- It has substantial internal density.
- Other modules already depend on it through a narrow surface.
- It can avoid depending back on its callers.

Preconditions for crate to module:

- The crate has low external fan-in.
- It exists mostly for build history or accident.
- Its public surface is not independently meaningful.

Preconditions for concrete to trait:

- More than one implementation is plausible, or test/mocking/dynamic dispatch has
  a real payoff.
- The callsite-usage set is smaller than the concrete type surface.
- The trait can be named as a contract, not as the original type with a suffix.

Checks:

- `crate_edges(directory=...)`
- `forbidden_dependency_check(directory=..., rules=[...])`
- `get_exports(...)`
- `dead_pub_in_crate(...)`
- `function_signature(target=...)`
- `who_uses(target=...)`
- `calls_from(caller=...)`

Failure modes:

- lifting a module into a crate before it has a stable surface,
- lowering a real bridge crate because it is small,
- extracting a trait that has only one possible implementer,
- adding wrappers that do not encode an invariant or reduce coupling.

## §6. Project Lifecycle

### L1. Starting Project

At the start, speed matters. Use agentic coding to discover:

- required features,
- app behavior,
- domain concepts,
- external dependencies,
- rough user workflows,
- failure cases.

The layout after this phase is expected to be bad. That is normal. The purpose
of the first phase is discovery, not architecture.

Common shape after rapid iteration:

- too many files in one directory,
- many directories with one file,
- unclear module names,
- broad imports,
- duplicated types,
- mixed infrastructure and domain code.

The first structural step is not blindly "create dirs." The first step is to
find dense groups. Then create shallow modules that reflect those groups.

Preferred early shape:

```text
src/dir1/file1.rs
src/dir1/file2.rs
src/dir2/file1.rs
```

Keep module depth to one directory where possible. This is an agent ergonomics
heuristic, not a law. A second level is justified only when the nested module has
its own dense internal graph and named surface.

Starting-project protocol:

1. Build a structural map.
   - `module_tree`
   - `get_dependencies` per file
   - `get_imports` per module
   - `who_imports` for candidate hubs

2. Identify dense groups.
   - files/modules with many mutual references,
   - files/modules that change or conceptually move together,
   - bridge units that many others import,
   - accidental junk drawers.

3. Propose a shallow module layout.
   - each directory gets one sentence,
   - each directory has more internal edges than external edges,
   - one-file directories are suspicious unless they are deliberate facades.

4. Apply Move and Split/Merge operations.
   - move files between module directories,
   - move functions/types between files,
   - split large mixed files,
   - merge tiny fictional modules.

5. Lift modules into crates only when the project size earns it.
   - target shape: `workspace/crate1/src/dir1/file1.rs`,
   - crate internals still prefer shallow module depth,
   - crate graph must be acyclic,
   - public surface must be narrower than internals.

6. After boundaries are stable, enforce local guidelines.
   - crate by crate,
   - module by module,
   - function by function only after the containing graph makes sense.

Guideline source:

- `.docs/rust-guidelines-final.md` when present in the project.
- Project instructions from `AGENTS.md` or equivalent local agent guidance.

Guideline tools:

- `unsafe_audit`
- `fn_body_audit`
- `recursion_check`
- `channel_capacity_audit`
- `derive_audit`
- `missing_docs_audit`
- `analyze_complexity`

Typical checks:

- no unjustified unsafe code,
- bounded loops and low cyclomatic complexity,
- small function/method bodies,
- small input arity,
- clear visibility,
- documented public surface,
- required standard derives.

Respect repository instructions while doing this work. In this repo, do not run
formatting.

### L2. Semi-Mature Project

In a semi-mature project, the structure is good enough that most features should
fit into an existing module or crate.

The main failure mode is feature scope. A feature is too large for an agent when
it requires simultaneous changes across too many unstable boundaries.

Agent rule:

Find the smallest implementation target with the highest leverage.

Best targets:

- one independent crate,
- one coherent module,
- one new file inside a coherent module,
- one local type/function cluster.

If no such target exists, redesign before implementing.

Redesign moves:

- wrap a type into a supertype,
- add a child/member type to represent a hidden invariant,
- split a type into two independent types,
- extract a trait from observed callsite usage,
- move a function/type to the module where its references already point,
- split a module whose public surface now says "and also."

Categorical diagnosis:

- The feature crosses too many morphisms.
- The target object is not isolated.
- The needed boundary does not exist yet.
- The existing quotient is fictional.

HoTT diagnosis:

- The type does not encode the feature's invariant.
- One type is carrying two propositions.
- A function has too many inputs because the domain shape is missing.
- Callers depend on concrete detail instead of a smaller contract.

### L3. Mature Project

In a mature project, the primary constraint is compatibility-preserving change.
The question is not only "what is the right design?" It is "how do we reach the
right design without breaking existing callers, stored data, migrations, tests,
or downstream crates?"

Mature-project operations:

- introduce new API beside old API,
- write adapters between old and new types,
- migrate callers incrementally,
- deprecate old surface after migration,
- preserve serialization and schema compatibility,
- gate risky redesigns behind feature flags,
- measure hub blast radius before touching public symbols.

Categorical reading:

- A migration is a mapping from the old dependency graph to the new dependency
  graph.
- External morphisms must keep landing somewhere valid during the migration.
- The old public surface can become a facade over the new graph.

HoTT reading:

- Old and new types need an equivalence or a deliberate one-way refinement.
- Adapters witness the path between old and new shapes.
- Tests witness that important paths commute: old caller behavior reaches the
  same result through the new implementation.

Useful mature-project checks:

- `recursive_callers_count` before changing a hub,
- `who_imports` and `who_uses_summary` before moving public items,
- `get_declared_reexports` before changing facade modules,
- `dead_pub_report` after migrations,
- `forbidden_dependency_check` after crate restructuring.

## §7. Diagnostics

Use these questions before choosing an operation.

| Question | Lens | Tools | Operation |
|---|---|---|---|
| Where do most references to this unit point? | Alignment / morphism density | `who_uses_summary`, `who_imports`, `get_dependencies` | Move |
| Does this boundary earn its existence? | Quotient quality | `get_imports`, `get_exports`, `module_tree` | Split/Merge or Lower |
| Is this container's public surface nameable? | Projection | `get_exports`, `get_declared_reexports`, `dead_pub_report` | Split/Merge |
| Would moving this create a cycle? | Acyclicity | `crate_edges`, `forbidden_dependency_check`, `calls_from` | Move or Lift blocked |
| Is this module large enough to be a crate? | Lift functor | `crate_edges`, `get_exports`, `who_imports` | Lift |
| Is this crate too small to remain a crate? | Lowering | `crate_edges`, `dead_pub_in_crate`, `who_imports` | Lower |
| Does this type carry multiple independent invariants? | Product/sum decomposition | `function_signature`, `who_uses`, source inspection | Split type |
| Is this trait honest? | Callsite-usage set | `who_uses`, `calls_from`, `function_signature` | Lift concrete to trait |
| Is this feature too large for one agent pass? | Boundary crossing | `who_uses_summary`, `module_tree`, proposal review | Redesign first |

Heuristics:

- A boundary is suspicious when external edges are comparable to internal edges.
- A directory with one file is suspicious unless it is a facade or future crate
  seed.
- A directory with many unrelated public items is a junk drawer.
- A crate with high fan-in and low fan-out is often a real core/domain crate.
- A crate with high fan-in and high fan-out is a bottleneck.
- A trait with one implementer and a full concrete-method surface is probably
  fake abstraction.
- A large feature with no isolated target is a design problem, not an
  implementation problem.

## §8. Operation Composition

Real refactors are compositions of the three primitives.

Examples:

### File to Module

1. Split one large file into several files.
2. Move related functions/types into those files.
3. Lift the file group into a directory module.
4. Project only the intended surface through `mod.rs` or the parent module.

Primitive sequence:

```text
Split -> Move -> Lift -> Project
```

### Module to Crate

1. Verify the module has a named public surface.
2. Verify most internal edges stay internal.
3. Lift the module directory into a crate.
4. Replace internal module paths with crate imports.
5. Verify `crate_edges` is acyclic.

Primitive sequence:

```text
Lift -> Move -> Project
```

### Crate to Module

1. Verify the crate has low independent fan-in.
2. Verify its public surface is not independently meaningful.
3. Lower the crate into a module under its main consumer.
4. Remove or shrink the crate dependency.
5. Verify no public facade was accidentally widened.

Primitive sequence:

```text
Lower -> Move -> Merge
```

### Concrete Type to Trait

1. Gather callsites.
2. Extract only the used contract.
3. Implement the trait for the concrete type.
4. Migrate callers that benefit from the abstraction.
5. Keep concrete callers concrete if they need concrete behavior.

Primitive sequence:

```text
Lift -> Project -> Move callers
```

### Type Split

1. Identify independent field or variant groups.
2. Create child/component types.
3. Move methods to the component that owns their invariants.
4. Add conversion or access paths only where callers need them.
5. Verify signatures got smaller or invariants got clearer.

Primitive sequence:

```text
Split -> Move -> Project
```

## §9. Forgetfulness

Most refactors are not reversible. This is already P9 in `THEORY.md`, but it is
central enough to repeat here.

Move can be mostly reversible when paths are preserved, but comments, imports,
and caller expectations still change.

Split/Merge is forgetful. After a merge, the old boundary is gone. After a
split, the original locality is gone.

Lift/Lower is strongly forgetful. Lifting to a trait forgets concrete detail.
Lowering from a trait back to a concrete type forgets why the trait surface was
chosen. Lifting a module into a crate creates a public API that callers may start
depending on.

Agent rule:

Before any forgetful operation, record the reason:

- which boundary was too expensive,
- which surface was too broad,
- which type invariant was missing,
- which callers justified the change,
- which tool output verified it.

## §11. Agent Output Format

For any structural or type-level refactor, the agent should report:

```text
Goal:
Target:
Maturity level: starting | semi-mature | mature
Primitive operation: Move | Split/Merge | Lift/Lower | composition
Category-theory reason:
HoTT/type reason:
Evidence:
Risk:
Verification:
```

Example:

```text
Goal: make indexing batching easier to modify
Target: src/indexing/embedding_batcher.rs
Maturity level: semi-mature
Primitive operation: Split + Move
Category-theory reason: batching and retry logic have denser internal edges than
their edges to the rest of indexing
HoTT/type reason: batch state and retry policy are separate product components
Evidence: get_dependencies, who_uses_summary, function_signature
Risk: medium, because indexing is a hub
Verification: tests, fn_body_audit, who_uses_summary after move
```

The explanation should cite the primitive operation and the evidence. Avoid
claiming a refactor is "cleaner" unless the graph or type shape explains why.

## §12. Short Rules

- Start messy to discover the app. Then refactor boundaries.
- Do structural refactors before local function cleanup.
- Prefer shallow module depth until deeper containment earns itself.
- A directory is a quotient of a graph, not a bucket for names.
- A crate is a lifted module with a public contract.
- Visibility is projection; do not widen it to make moves easy.
- A trait is the contract callers use, not the concrete type copied sideways.
- Split when independent surfaces exist.
- Merge when the boundary is fictional.
- Lift when a stable public surface exists.
- Lower when the higher-level boundary has no independent job.
- In mature code, migrate through adapters instead of breaking callers.
- Record evidence before forgetful operations.
